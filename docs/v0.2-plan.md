# v0.2 implementation plan

> **Status:** plan, not contract. The binding contract is
> [`cli-design.md`](./cli-design.md). This document sequences the work
> needed to ship v0.2 of that contract; it doesn't redefine it. If a
> conflict arises, `cli-design.md` wins and this plan is updated.

## 1. Scope summary

v0.2 is "**the mutating core — agents can drive a backlog**". v0.1
gave agents the read surface plus four narrow mutation verbs (`item
set/clear/update`, `update create`); v0.2 lifts the write surface to
parity with the read surface so an agent can create items, move them
between groups and boards, archive/delete them, post and edit
comment threads, and shape the boards themselves. The friendly
column-value translator gains six more types, plus the long-deferred
`--set-raw` escape hatch.

**In scope (from `cli-design.md` §13 v0.2):**

| Group | Commands |
|-------|----------|
| **column writers** | `--set-raw <col>=<json>` escape hatch + friendly translators for `link`, `email`, `phone`, `tags`†, `board_relation`†, `dependency`† (†tentative — may slip to v0.3 if fixture work surfaces blocking design cost; see §6 risk register) |
| `item` | `create`, `archive`, `delete`, `duplicate`, `move`, `upsert`, bulk `clear --where ... <col>` |
| `update` | `reply`, `edit`, `delete`, `like`, `unlike`, `pin`, `unpin`, `clear-all`; `list --board <bid>`; `list <iid> --with-replies` |
| `workspace` | `create`, `update`, `delete`, `add-users`, `remove-users` |
| `board` | `create`, `update`, `archive`, `delete`, `duplicate`, `add-users` |
| `board` (structure) | `column-create`, `column-update`, `column-delete`, `group-create`, `group-update`, `group-archive`, `group-duplicate`, `group-delete` |
| polish | NDJSON streaming for `item search` and `update list`; refreshed envelope-snapshot suite; `0.2.0` release artefacts |

**Universal partial-success rule (cli-design §6.1 compliant).** Multi-
target verbs (`workspace add-users` / `remove-users`,
`board add-users`, `update clear-all`) emit one **success envelope**
(`ok: true`) with per-target results in `data.results: [...]`. Each
record carries its own `{<target_id_field>, ok: true | false,
error?}` where the field name is verb-specific — `user_id` for
add-users / remove-users, `update_id` for `clear-all`, `<verb>_id`
generally. The shared envelope-builder (`buildPartialSuccessMutation`)
takes the field name as a parameter; the per-record shape is
otherwise uniform (same `ok` and optional `error: { code, message }`).
The envelope's top-level `error` slot is reserved for whole-call
failure (couldn't reach the API, couldn't resolve any target before
the loop began, an unrecoverable validation error). All-failed-but-
each-call-attempted is still `ok: true` — the agent reads
`data.results` to determine outcomes. This avoids a §6.1 contract
revision (Codex round-1 P1).

**Out of scope (deferred per §13 "Explicitly deferred"):**

- `monday dev …` — Monday Dev workflow shortcuts (v0.3).
- `monday auth login` / OAuth profiles / config files / `~/.monday-cli/config.toml` (v0.3).
- `monday status` / `monday usage` diagnostics (v0.3).
- `notification send` / `webhook list/create/delete` (v0.3).
- `item search` cross-board (omit `--board`); `item history`; `board favorites`; `item update --continue-on-error` partial-success bulk (v0.3).
- `item watch`; `--concurrency`; asset upload (`add_file_to_column`); shell completion; `doc list/get`; `team` create/manage (v0.4).
- Boolean filter DSL — v0.2 keeps the v0.1 `--where` operator allowlist + `--filter-json`. The boolean DSL is fixture-driven; only ships when adoption metrics show concrete demand. Pencilled at v0.2 stretch / v0.3 firm.
- `monday item recreate-from-archive` — still no misleading `restore` command (cli-design §5.4 stands).

## 2. Pre-flight: what's already in place

Working at HEAD (`main`, post-`638d49a`):

- **Foundations (M0–M2.5)** — the runner / signal handling / typed errors / redaction / envelope builders / commander wiring / transport seam / retry layer / `MondayClient` / `account` commands all live. v0.2 inherits these unchanged.
- **Read surface (M3–M4)** — `workspace`, `board`, `user`, `update`, `item` reads (incl. filter DSL, cursor pagination, NDJSON streaming for `item list`, projection layer). Every write verb in v0.2 leans on these for resolution: `--board <bid>` lookup, column resolution, `me` token, board metadata cache.
- **Write half of §5.3 (M5a)** — `column-values.ts` (translators for `text`/`long_text`/`numbers`/`status`/`dropdown`/`date`/`people`), `dates.ts`, `people.ts`, `column-types.ts` writable allowlist, `dry-run.ts` engine. **M8's writer expansion plugs into this** — six new translator slots in the same dispatcher, plus the long-deferred `--set-raw` path.
- **Mutation surface (M5b)** — `item set`/`clear`/`update` (single + bulk via `--where`) + `update create`. Shared helpers: `item-helpers.ts`, `resolver-error-fold.ts`, `resolveColumnsAcrossClauses`, `parse-boundary.ts`. **M9–M13 reuse these directly**: bulk `item clear --where` is the per-item leg already plus a cursor walk; `item create`/`upsert`/`move` reuse the resolver fold; `update reply`/`edit`/`delete` reuse `update create`'s body-source plumbing (`--body` / `--body-file` / stdin).
- **M9.5 cleanup helpers (post-M9, pre-M10)** — six new shared modules + one drift fix collapse ~1900 LOC of duplication: `set-expression.ts splitSetExpression` (R22), `resolution-context.ts buildResolutionContexts` (R24), `item-board-lookup.ts` (R23 — `lookupItemBoard` + `lookupItemBoardWithHierarchy` + `resolveBoardId`), `source-aggregator.ts mergeSource/mergeSourceWithPreflight/mergeCacheAge` (R21), `resolver-error-fold.ts foldAndRemap` extension (R26 — mutation catch-arm wrapper), `resolution-pass.ts resolveAndTranslate` (R20 — three-pass column resolution helper). **M10–M18 inherit these on day one**: each new mutation command (archive / delete / duplicate / move / upsert) starts with `resolveBoardId` + `foldAndRemap`; column-write surfaces inherit `resolveAndTranslate` + `buildResolutionContexts`. The duplication-trigger threshold is "lift on the third consumer" per v0.1-plan §17 R-timing rule; ALL future write surfaces are now consumer N+1, never N=1, of these patterns.
- **Diagnostics + escape hatch (M6)** — `board doctor`, `raw`, agent-flow E2E. `raw` already gives agents a write escape; v0.2's `--set-raw` is the typed alternative for the column-value path specifically.
- **Release scaffolding (M7)** — `published-tarball.test.ts`, `envelope-snapshots.test.ts` (40 snapshots), `output-shapes.md`, `CHANGELOG.md`. **M18's release prep extends each**: ~30 new envelope snapshots, ~30 new `data` shape entries, `0.2.0` changelog cluster.
- **Test infrastructure** — fixture transport, fixture server (E2E), redaction-hardening suite, agent-flow E2E. Coverage floor sits at **94% branches / 95% lines / functions / statements**; v0.2 holds or raises it (never lowers).
- **Tooling** — ESM, NodeNext, strictest TS, ESLint `strictTypeChecked + stylisticTypeChecked`, vitest 4. CI runs typecheck + lint + test:coverage + build smoke on Node 22 and 24. Two-AI review (Codex `gpt-5.5`, read-only) per design change and per implementation milestone.

What's **missing** that v0.2 needs (each filled by the named milestone):

- A **cache-invalidation discipline** for board-metadata-mutating commands (M16). Today, every `--set` consults the cached board metadata; mutations that change board shape (`column-create`/`column-delete`/`group-rename`) need to invalidate the cache so subsequent reads/writes see the new shape. v0.1 had no schema-mutating commands so the question never arose.
- A **partial-success envelope** for multi-target add-users / remove-users (M14, M15). Single-target failure is the v0.1 mutation envelope; for these multi-user ops, per-user `{ok, error?}` records keep the agent informed without abandoning the whole call. Fail-fast bulk update (cli-design §6.5 `details.applied_to`) is the working precedent — `--continue-on-error` for `item update` waits until v0.3.
- A **`--set-raw <col>=<json>`** path through the existing translator (M8). The translator currently dispatches on type; raw bypasses dispatch but still resolves the column ID and validates the JSON shape against zod.
- A **column-mapping JSON shape** for cross-board `item move` (M11). Simple `{<source_col_id>: <target_col_id>}` (string-to-string) mapping that translates one-for-one onto Monday's `columns_mapping: [ColumnMappingInput!]` parameter. The richer `{id, value?}` value-translation form is deferred to v0.3 (Monday's wire shape carries no value slot — supporting it requires a non-atomic post-move mutation; see §15 M11 post-mortem).
- An **idempotency layer** for `item upsert` (M12). Match-by lookup + create-or-update branching; the branch decision is captured in the mutation envelope (`operation: "create_item" | "update_item"`) so agents can reason about which path fired.
- A **board-structure mutation cluster** (M16). Column/group create/update/archive/delete each have one Monday mutation; what's new is the cache invalidation fan-out + the column-type allowlist for `column-create`.

## 3. Milestone sequencing

Eleven milestones (M8–M18; M16 split into columns-then-groups per
Codex P2). Each milestone is a coherent commit cluster (3–10 commits)
ending on a green tree with the listed exit criteria met. Earlier
milestones unblock later ones; do not start M(N) until M(N-1)'s exit
criteria are satisfied. Each milestone updates `docs/output-shapes.md`
for any new command shapes it lands and adds envelope snapshots for
any new emitted shape.

Order at a glance: **writer expansion → item lifecycle → bulk symmetry
+ idempotency → updates → workspace + board CRUD → board columns
(introduces eager invalidation contract) → board groups (reuses it)
→ polish + release**.

```
       M8                                 — writer expansion (foundations)
       M9 ─ M10 ─ M11                     — item lifecycle (create / archive+delete+duplicate / move)
       M12                                — item upsert + bulk item clear
       M13                                — update mutations + update list expansions
       M14 ─ M15                          — workspace + board lifecycle
       M16                                — board columns (cache-invalidating; also pins the eager-invalidation contract)
       M17                                — board groups (cache-invalidating, smaller surface)
       M18                                — NDJSON streaming + 0.2.0 release prep
```

### M8 — Writer expansion + `--set-raw` escape hatch

The foundational v0.2 milestone. No new commands ship — instead,
the column-value translator gains the firm-row translator slots and
the `--set-raw` escape hatch, so every later milestone (M9 `item
create`, M11 `item move --columns-mapping`, M12 `item upsert`,
M16 `column-create`) can lean on the expanded writable type surface
without re-touching tests. **Foundations-first** is the load-bearing
ordering decision (§8 decision 1).

**As shipped (M8 close).** Three firm-row friendly translators
landed (`link` / `email` / `phone`). `WRITABLE_COLUMN_TYPES` grew
from 7 to 10. The three tentative-row types (`tags` /
`board_relation` / `dependency`) deferred to a follow-up; the
runtime keeps them in `V0_2_WRITER_EXPANSION_TYPES` so
`unsupported_column_type` still carries `deferred_to: "v0.2"` and
points agents at `--set-raw` as the working escape hatch in the
meantime. The §16 M8 close-pass Codex review caught six P2 items
(see §10 post-mortem); fixes for cumulative warning preservation,
resolution-before-translation ordering, bulk cache-age aggregation,
and argv-parse-before-board-lookup landed in the same commit
sweep.

**Deliverables**
- `src/api/column-values.ts` — three new firm translator slots:
  - `link` — `<url>` (one segment) → `{url:<url>, text:<url>}`; `<url>|<text>` (pipe-split, max 1 split, both segments trimmed) → `{url, text}`. `text` defaults to the URL when the pipe form is absent. URL validated via `z.string().url()`; failure → `usage_error`.
  - `email` — single email → `{email:<value>, text:<value>}`; `<email>|<text>` → `{email, text}`. Email validated via `z.string().email()`; failure → `usage_error`.
  - `phone` — `<phone>|<country>` (pipe form mandatory) → `{phone:<phone>, countryShortName:<country>}` where `<country>` is a 2-letter ISO 3166-1 alpha-2 code (uppercase). E.164-loose validation (`+?\d{6,15}`); ISO code validated against a frozen allowlist (~250 codes from `Intl` / a vendored list). Single-segment `<phone>` (no `|<country>`) rejected with `usage_error` — Monday's phone-column validation requires both number and country code AND verifies they match (per Monday's phone-validation changelog), so the friendly translator can't safely default `countryShortName: ""`. Agents who need empty-country phones use `--set-raw`.
  - **Deferred (still tentative; `V0_2_WRITER_EXPANSION_TYPES`):** `tags` (account-tag lookup; adds `tag-directory.ts` mirroring `user-directory` from M5a; new error code `tag_not_found` only if it ships firm — §8 decision 2), `board_relation` (comma-split item IDs → `{item_ids:[N1,N2]}`; allowed-boards check against `settings.boardIds ?? [settings.boardId]`), `dependency` (same shape as `board_relation` against Monday's separate `dependency` payload). All three can be unblocked by `--set-raw` today; landing the friendly translators is a follow-up M8x or rolls into v0.3.
- `src/api/raw-write.ts` — the `--set-raw <col>=<json>` path. Resolves `<col>` through the same `resolveColumnWithRefresh` flow as `--set`. Two reject lists fire after resolution but before mutation (matching cli-design §5.3 escape-hatch contract):
  - **Read-only-forever** (`mirror`, `formula`, `auto_number`, `creation_log`, `last_updated`, `item_id`) → `unsupported_column_type` with `read_only: true`.
  - **`files`-shaped** (`file`, anything else where Monday's write path is `add_file_to_column` rather than `change_column_value`) → `unsupported_column_type` with `deferred_to: "v0.4"`.
  Otherwise validates `<json>` parses as a `JsonObject`, then constructs a `TranslatedColumnValue` with `payload: { format: 'rich', value: <parsed> }` so the existing `selectMutation` dispatcher handles it uniformly. **No type-shape validation** beyond the JSON-object boundary — the user took the escape hatch and owns wire-shape correctness; Monday's server-side rejection surfaces as `validation_failed`. Mutually exclusive with `--set` for the same `<col>` (resolution-time enforced — the duplicate-ID check fires after both flags' tokens resolve to column IDs, since argv-parse can't tell whether two distinct tokens alias to the same column without board metadata; cli-design round-2 P2 correction). **Pass ordering**: M8 close-pass refactor (§10 post-mortem item #2) split each command's resolution path into resolve-all → cross-token-duplicate-check → translate, so a friendly+raw pair targeting the same resolved column surfaces the mutual-exclusion `usage_error` even when the friendly value would have produced a translator error.
- `src/api/column-types.ts` — `WRITABLE_COLUMN_TYPES` grows from 7 to 10 (the three firm rows). The frozen `as const` array's order is part of the contract; tests iterate it. The three tentative types stay outside the writable allowlist via `V0_2_WRITER_EXPANSION_TYPES` so the `unsupported_column_type` branch still fires with `deferred_to: "v0.2"` for them. Update `commands/board/describe.ts` so `example_set` covers the firm new types.
- `unsupported_column_type` — for v0.1's `link`/`email`/`phone` row, `deferred_to: "v0.2"` disappears (now writable). The `tags`/`board_relation`/`dependency` row keeps `deferred_to: "v0.2"` (a follow-up promotes them or rolls them to v0.3). Read-only-forever and `deferred_to: "future"` rows unchanged.
- `monday item set` and `monday item update` learn `--set-raw <col>=<json>`. Bulk `item update --where` learns `--set-raw` too. `--set` and `--set-raw` against the same column are mutually exclusive (resolution-time enforced; `usage_error`). Help text gets an example block.
- `cli-design.md` §5.3 step 4 + writer-expansion roadmap table + §4.3 command-tree (adds `--set-raw` to the relevant signatures) updated to reflect the v0.2 firm rows. M8 ships its own design diff for the design doc — submitted alongside the M8 PR for two-AI review per CLAUDE.md.

**Tests (pyramid)**
- Unit: each new translator's grammar — happy path, every error path, Unicode in the human-readable text segment, pipe-split edge cases (empty trailer, multiple pipes, leading/trailing whitespace), idempotent translation (re-running translation on the output reproduces the wire shape). Wire-shape fixture pinned per (count × type) cell so v0.3 inherits the rule unchanged. `--set-raw` coverage: column resolution succeeds, JSON parse fails (`usage_error`), JSON parse succeeds and round-trips through `selectMutation`, `--set` and `--set-raw` against the same column collide (`usage_error`).
- Integration: `item set --board <bid> <link-col>=https://example.com|Site` produces the expected wire payload (cassette pinned). Repeat for each new type. `--set-raw` against an `unsupported` / `future` column type that accepts `change_column_value` *succeeds* at the CLI layer (the escape hatch's whole point) and surfaces Monday's rejection as `validation_failed`. `--set-raw` against read-only-forever (`mirror` / `formula` / etc.) or `files`-shaped (`file`) → `unsupported_column_type` at the CLI layer, no API call fires.
- E2E: one happy-path spawn per new firm type (`link`, `email`, `phone`); the tentative types ship with E2E too if they make the cut.

**Exit**
- `monday item set <iid> --board <bid> link_col=https://x` succeeds against fixtures.
- `monday item set <iid> --board <bid> --set-raw status='{"label":"Done"}'` succeeds against fixtures and equals the friendly `--set status=Done` payload byte-for-byte.
- `unsupported_column_type` `deferred_to: "v0.2"` no longer fires for any of the firm v0.2 row types.
- `board describe`'s `example_set` covers all newly-writable types.
- Two-AI review pass cleared.
- Coverage floor held at 94/95/95/95.

### M9 — `item create` (incl. subitem creation + position)

The first net-new item-mutation verb. Non-idempotent (creating an
item with the same name twice yields two items — cli-design §5.8);
ships with `--dry-run`.

**Deliverables**
- `src/commands/item/create.ts` — `monday item create --board <bid> --name <n> [--group <gid>] [--set <col>=<val>]... [--set-raw <col>=<json>]... [--parent <iid>] [--position before|after --relative-to <iid>] [--dry-run]`. Calls `create_item` (or `create_subitem` when `--parent` is present) with the full `--set` payload bundled into `column_values`. **Single round-trip is a hard exit gate** (Codex P1): if Monday's `create_item.column_values` proves not to accept any v0.2 type's wire shape, M9 halts; we don't fall back to `create_item` + `change_multiple_column_values` because a partial-state failure between the two calls leaves an item with the requested name but missing column values. The v0.1 fixture pass against `column_values` covers all seven v0.1 types; M8's three firm new types (`link`/`email`/`phone`) each get a fixture-pinned wire-shape test on the create path before M9 implementation begins. The three deferred types (`tags`/`board_relation`/`dependency`) only need `create_item` fixtures once their friendly translators ship. `--position` resolves to Monday's `position_relative_method` + `relative_to`.
- `--parent <iid>` triggers `create_subitem`. Subitems live on Monday's auto-generated subitems board; column resolution targets *that* board, not the parent's. M9 plumbs the resolved subitem-board ID through the same column-resolution path as the top-level path. **Classic boards only.** Multi-level boards (`hierarchy_type: "multi_level"` per cli-design §2.8 — where subitems live on the parent's board rather than an auto-generated subitems board) are rejected pre-mutation with `usage_error` carrying `details.hierarchy_type` and a hint pointing at v0.3. Per Codex round-1 P1 on the design diff: M9's column-resolution path assumes the auto-generated-subitems-board model and would resolve against the wrong metadata on multi-level boards. Multi-level subitem support is a v0.3 candidate that needs its own design pass + cli-design §2.8 widening + a hierarchy-type-aware column-resolution branch. The pre-mutation hierarchy_type check happens after parent-item lookup (the CLI needs the parent's `board.hierarchy_type` to gate) but before any column resolution / value translation / mutation.
- `--position before|after --relative-to <iid>` — both flags required together; one without the other is `usage_error`. `relative_to` must reference an item on the same board (verified by item lookup, mirrors M5b's wrong-board check).
- `idempotent: false` in the CommandModule. Help text calls out `item upsert` (M12) for the idempotent variant.
- Mutation envelope: `data: { id: <new_item_id>, name, board_id, group_id, parent_id? }` plus the v0.1 top-level `resolved_ids: { ... }` slot per cli-design §6.4 (Codex round-2 P2: v0.1 mutation contract uses top-level `resolved_ids`, not nested in `data`). Snapshot pinned.
- Resolver-warning fold: every `--set`-resolution warning that surfaces during the create flow folds into `details.resolver_warnings` on failure (mirrors M5b discipline).

**Tests (pyramid)**
- Unit: argv parser — `--position` requires `--relative-to`; `--parent` and `--group` are mutually exclusive (subitems live on the subitems board, not in groups); `--name` empty after trim → `usage_error`; multiple `--set` against the same column → `usage_error`.
- Integration: happy path (top-level item, default group, multiple `--set`); subitem path (`--parent <iid>`); position path (before / after); each error path (`not_found` on board, `not_found` on parent, `not_found` on relative-to, `validation_failed` on Monday rejection); dry-run path **for both branches** — top-level `operation: "create_item"` AND subitem `operation: "create_subitem"` envelopes pinned (Codex round-4 P2 — both planned-change operations are §9 preconditions; tests must assert both shapes).
- E2E: one happy-path spawn (top-level), one subitem spawn.

**Exit**
- `monday item create --board <bid> --name "Test" --set status=Done` returns a new item ID against fixtures.
- Subitem path verified (top-level create + subitem create round-trip).
- `--dry-run` echoes the planned create without firing the mutation.

### M10 — `item archive` + `item delete` + `item duplicate`

Lifecycle siblings. Share the confirmation-gate pattern from M5b's
bulk paths (`--yes` required for destructive verbs).

**Deliverables**
- `src/commands/item/archive.ts` — `monday item archive <iid> --yes [--dry-run]`. Calls `archive_item`. `--yes` required for live archive (destructive — archived items hide from default queries and Monday only retains them for a 30-day recovery window). cli-design §4.3 line 509 currently shows `item archive <iid>` without `--yes`; §9 precondition adds the `--yes` to the command-tree before M10 ships. `--dry-run` reports the would-archive item state. Idempotent on the wire (re-archiving an archived item is a no-op on Monday's side); CLI marks `idempotent: true`.
- `src/commands/item/delete.ts` — `monday item delete <iid> --yes [--dry-run]`. Calls `delete_item`. `--yes` mandatory (truly destructive — Monday's delete is recoverable for 30 days but the CLI doesn't promise that). Not idempotent (re-deleting an already-deleted item is `not_found`).
- `src/commands/item/duplicate.ts` — `monday item duplicate <iid> [--with-updates] [--dry-run]`. Calls `duplicate_item` with `with_updates` boolean. Returns the new item's ID + the source item's ID. **Not idempotent** — every call creates a new item.
- All three share `ITEM_FIELDS_FRAGMENT` / `parseRawItem` / `projectItem` (the v0.1 R9 + R18 boundary) so every envelope's `data` (live) or `planned_changes[0].item` (dry-run) matches `monday item get` byte-for-byte. Dry-runs do a single dedicated read (`ItemArchiveRead` / `ItemDeleteRead` / `ItemDuplicateRead`) — `runByIdLookup` would be the wrong helper here because the cli-design §6.4 dry-run shape needs `data: null` + `planned_changes`, not the get-shaped envelope `runByIdLookup` emits. **`meta.source` is always `"live"`** for both paths: archive's + delete's live mutations are single-leg wire calls; duplicate's live is two-leg (`ItemBoardLookup` + `duplicate_item` — Monday requires `board_id`, derived from the source item's board); duplicate's dry-run is single-leg (`ItemDuplicateRead`). No leg consults the cache — the wire calls are unconditional — so `mergeSource` doesn't appear and `cache_age_seconds: null`.

**Tests (pyramid)**
- Unit: `--yes` enforcement on `archive`/`delete`; `--with-updates` boolean parser; idempotency-flag in CommandModule shape.
- Integration: each verb happy path; each error path (`not_found`, `forbidden`, `validation_failed`); `confirmation_required` without `--yes` for `delete`; dry-run for each.
- E2E: one happy-path spawn for `delete --yes` (it's the highest-blast-radius verb); the tarball test learns the new commands.

**Exit**
- All three verbs return correct envelopes against fixtures.
- `monday item delete <iid>` without `--yes` exits 1 with `confirmation_required`.

## 4. Cross-cutting "definition of done" per command

v0.2 inherits v0.1's §4 checklist verbatim with four additions:

12. **Cache invalidation discipline.** Any command that mutates
    board metadata — column or group shape (M16, M17) — calls
    `invalidateBoard(boardId)` after a successful response. The
    v0.1 board-metadata cache includes both `columns: [...]` and
    `groups: [...]`; either kind of mutation invalidates. Documented
    in the command's source header. The integration test exercises
    a `mutate → read → see fresh state` round-trip AND asserts the
    cache-miss-refresh path did not fire (eager invalidation, not
    lazy refresh).
13. **Partial-success envelope shape pinned.** Multi-target commands
    that emit per-target results (`workspace add-users` /
    `remove-users`, `board add-users`, `update clear-all`) document
    the `data.results: [...]` slot in `output-shapes.md` and pin
    one envelope snapshot per verb. Envelope is always `ok: true`
    when dispatch ran (per §1 universal rule); top-level `error`
    only on whole-call failure.
14. **Every mutation verb supports `--dry-run`.** cli-design §9.2
    is binding — there are no exceptions. Verbs whose underlying
    Monday call isn't itself a column-value mutation (e.g.
    `archive_item`, `delete_workspace`, `like_update`) get a
    minimal dry-run shape: `operation: "<mutation_name>"` plus the
    primary IDs, no diff section. cli-design §6.4 receives a
    design diff per milestone listing the new `planned_changes.
    operation` values (§9 preconditions).
15. **Idempotency reasoning written down.** Every M9–M17 command's
    source header has a one-line idempotency rationale (yes / no /
    "yes when X branch fires") so a future agent reading the file
    doesn't have to reverse-engineer it from cli-design.md.

## 8. Decisions (open questions to close before M8)

These questions are open at v0.2-plan draft time. Each gets a decision
recorded here before M8 starts so implementation doesn't relitigate.
The two-AI review on this plan should land each one.

1. **Foundations-first vs agent-value-first ordering.** Closed: foundations-first (M8 before any new mutation). Re-touching M9–M17 when M8 lands later is roughly 5 milestones × test-suite extension cost; foundations-first eats one milestone of "no visible progress" but saves the rework. The visible-progress alternative (M9 first, M8 inserted later) is rejected.
2. **Stable error code count grows from 26 to 27 or 28.** Closed: `tag_not_found` (M8, only if `tags` ships firm — adds 1 to v0.1's 26), `ambiguous_match` (M12 — adds 1 unconditionally). Total: 27 if `tags` slips, 28 if both ship. A 29th slot reserved for a possible `cross_board_columns_unmatched` if M11's `usage_error` decoration proves insufficient — pencilled, not committed.
3. **Partial-success envelope shape — `data.results: [...]` vs new top-level `results`.** Closed: `data.results: [...]` reuses the existing mutation envelope. **The envelope is always `ok: true` when dispatch ran**, even when every per-target call inside the loop failed — the call's contract is "I ran the dispatch and here are the per-target outcomes." Top-level `error` (`ok: false`) is reserved for whole-call failure (couldn't reach API, couldn't resolve any `--users` token before the loop began, etc.). This avoids a §6.1 contract revision (Codex round-1 P1: my draft incorrectly said `ok: false` when all targets fail). Per-target record shape: `{<verb_target_id>: string, ok: true | false, error?: { code, message }}` where the id field is `user_id` for add-users / remove-users and `update_id` for `clear-all`. The shared `buildPartialSuccessMutation` helper accepts the field name as a parameter so the per-verb shape is self-documenting in JSON without forcing every consumer to adopt a generic `target_id`.
4. **`item upsert --match-by` accepts `name` or only column tokens?** Closed: accepts `name` as a special token (matches `item.name`). Justification: `name` is the most common match-by case for "is this item already there?" and forcing agents to first map name to a column ID feels wrong. Documented in cli-design §5.8 update.
5. **Cross-board `item move` default — strict (require mapping) or permissive (drop unmatched)?** Closed: strict. Monday's permissive default silently drops column values; the CLI rejects with `usage_error` and a `--columns-mapping` example. `--columns-mapping {}` (empty object) is the explicit "drop everything" opt-in.
6. **Cache invalidation — eager or lazy?** Closed: eager. Each M16 column-mutation calls `invalidateBoard(boardId)` post-success. The cache-miss-refresh path (cli-design §8) stays as a backstop for cache-state drift between processes (one CLI invocation invalidates the cache; a parallel CLI invocation in another process sees the stale cache until its own call). Cross-process invalidation is a v0.3 candidate. Requires a cli-design §8 design diff (§9 precondition).
7. **`column-create --type` allowlist enforcement.** Closed: validate against known Monday column types (any of the ~40-type set), warn (don't block) when outside `WRITABLE_COLUMN_TYPES`. Warning is **category-branched** per cli-design §5.3 escape-hatch contract — raw-writable types suggest `--set-raw`; read-only-forever flag that there's no write path at all; `files`-shaped types flag the v0.4 deferral. Blocking would be paternalistic — agents may legitimately create read-only columns.
8. **`update clear-all` — sequential delete or parallel?** Closed: sequential (mirrors M5b bulk update). Parallel waits for the v0.4 `--concurrency` flag.
9. **Archive `--yes` consistency across nouns.** Closed: every archive verb requires `--yes`. cli-design §3.1 says destructive ops without `--yes` return `confirmation_required`; archive hides items/boards/groups and only Monday's UI can recover them within ~30 days, so it counts as destructive. cli-design §4.3 line 509 currently shows `item archive` without `--yes` — §9 precondition adds it. Same for `group-archive`. Codex P1 caught the inconsistency in my v0.2-plan first draft.
10. **M16 split — columns first, then groups.** Closed: M16 ships columns (`column-create/update/delete`) and **introduces the eager-invalidation contract** + `invalidateBoard(boardId)` helper. M17 ships groups (`group-create/update/archive/duplicate/delete`) and **reuses** the M16 invalidation contract — both column and group mutations invalidate the v0.1 board-metadata cache (the cache holds both `columns: [...]` and `groups: [...]`; an earlier draft of this decision claimed groups had no cache pressure — Codex round-2 P1 caught the misclaim). M18 absorbs what was the old M17 (NDJSON + release prep). Justification: cache invalidation is the highest-risk design surface in board-structure mutations; concentrating the *contract introduction* in M16 with three verbs keeps the round-trip tests reviewable. Codex round-1 P2 caught the over-stuffed M16 in the first draft.
11. **Every mutation supports `--dry-run` — no exceptions.** Closed: cli-design §9.2 is binding. Verbs whose underlying Monday call isn't a column-value mutation (e.g. `archive_item`, `delete_workspace`, `like_update`) get a minimal dry-run shape: `operation: "<mutation_name>"` plus the primary IDs, no diff section. cli-design §6.4 receives a per-milestone design diff listing the new `planned_changes.operation` values (§9 preconditions). Codex P1 caught the omission in my first draft.

## 10. M8 post-mortem — what Codex caught after implementation

### Findings table

| Code | Severity | Area | Finding |
|------|----------|------|---------|
| F1 | P2 | dry-run engine | `--set-raw` translator catch folded only the current resolution's warnings, dropping cumulative warnings from prior `--set` legs. M5b R19 contract violation. |
| F2 | P2 | resolution ordering | Translation ran inline with resolution across dry-run + live single + live bulk. `--set X=bad-value --set-raw X={...}` surfaced the translator `usage_error` instead of the cli-design §5.3 line 961-972 mutual-exclusion `usage_error`. |

| F4 | P2 | argv-parse ordering | `item update` single resolved board ID via `ItemBoardLookup` BEFORE parsing `--set-raw`. Without `--board`, malformed JSON paid a network round-trip — diverged from `item set` and the bulk path's argv-boundary contract. |
| F5 | P2 | docs drift | `docs/v0.2-plan.md` §3 M8 still described the pre-slip scope (six translators, `WRITABLE_COLUMN_TYPES` → 13, tentatives slipping to v0.3). Reality: 3 firm types shipped, total 10 writable, tentatives still `v0.2`. |

| P3-1 | P3 | comments | `column-values.ts` module header still said v0.1 had no `--set-raw` and listed link/email/phone in the unsupported v0.2 branch. Stale post-M8. |
| P3-2 | P3 | comments | `board doctor`'s `v0_2_writer_expansion` message said v0.2 "will add it (plus --set-raw)" — `--set-raw` already exists post-M8. |

### What pattern explains the findings

All six P2s cluster around a single failure mode: **multiple legs
that should compose coherently were composed by inlining the
translation step into the resolution loop.** The clean fix in F2
(separate resolve / cross-token-check / translate passes) makes
F1 fall out for free (each catch site folds the cumulative
`warnings` array, not the per-token slice) and reshapes the
update.ts paths into three parallel implementations of the same
contract.

F3 is the same shape one layer up: meta-resolution ran live, then
per-token resolution ran cache, and the aggregate logic only
tracked source — not age. The fix mirrors the single-item path's
`aggregateCacheAge` running max.

F4 and F5 are scope-of-fail-fast bugs: the argv-boundary contract
("malformed input fails before any network call") wasn't applied
uniformly across `item update`'s single path; the doc drift was
the v0.2-plan being written for the pre-slip scope and never
revised when the M8 implementation slipped tags / board_relation
/ dependency to a follow-up.

### Single load-bearing lesson

**Resolve everything before translating anything.** A multi-token
mutation surface has three implicit phases: (1) resolve every
column token to a real column ID (cache, refresh, archived check,
warnings), (2) detect duplicate-target collisions across tokens
(same-token, same-resolved-id), (3) translate each input value
into a wire payload. M5b's single-token surfaces conflated
phases 1 and 3 because there was nothing to compare against;
M8's two-flag surface (`--set` + `--set-raw`) made the
conflation visible because the same column can be addressed two
ways and the cross-token check needs both legs resolved before
it can fire.

This is the same lesson v0.1's M3 logged about cache + live
provenance ("aggregate source after every leg, never before"),
generalised to "aggregate every reusable property after every
leg, then act on the aggregate." For M9 (`item create`) and
later mutation surfaces with column payloads: keep resolution
and translation as separate passes by default; if a future
optimisation collapses them, fence the collapse with explicit
asserts that the cross-token duplicate check still fires
pre-translation.

### What pattern explains the findings

Two patterns, both familiar from M0 / M5b / M8 post-mortems:

**Pattern 1 — "the argument that the gate is unnecessary."** P1 (#1)
came from a deliberate decision to skip the `validation_failed` →
`column_archived` remap on create. The reasoning seemed solid:
"the resolver already gates on archived (`includeArchived: true` +
explicit throw), so a Monday `validation_failed` here is a true
value-shape failure, not a stale-cache miss." But the cache *can*
say active when Monday has since archived — that's the exact
scenario the F4 helper exists for. M5b ran into a cousin of this
when bulk failures dropped the remap (M5b finding #3, fixed in
M5b cleanup re-review #2). The lesson: when a contract helper
exists across multiple call sites (item set, item update single +
bulk, and now item create), the default is to wire it through;
the burden is on the caller to argue why it shouldn't apply, and
"the gate above caught everything I thought of" is rarely a
strong enough argument.

**Pattern 2 — "match_operation_name covers wire shape."** P2 (#3)
was M8's exact lesson recurring: integration tests that match only
on `operation_name` accept any wire payload, so a regression in
`columnValues` shape, position enum mapping, or parent-id routing
slips through. M8's fix-up commit wired `match_variables` across
item set / item update fixtures; M9 inherited the testing helpers
but didn't inherit the discipline. The round-2 P2 follow-up (long_text
re-wrap pin) is a sub-case of the same pattern — even with
`match_variables` on most fixtures, missing one column type (long_text)
leaves the most-divergent wire shape unpinned. Lesson: when
`bundleColumnValues` (or any shared shape projection) has a
type-specific branch, every type with a divergence needs its own
fixture pin, not just the common case.

**Sub-pattern — "the envelope source aggregator never lies twice."**
P2 #1 (preflight source aggregation) was the third instance of "the
envelope source must reflect every wire leg that fired" pattern
post-M3. M3's original error was inferring source from warnings
alone (cache hit → live envelope). M5b's bulk path repeated it on
the cache-age slot. M9's twist was a network leg that ran *before*
the dry-run engine (parent lookup) — the engine's `source: 'none'`
short-circuit is honest about the engine's own legs but knows
nothing about pre-planner work. Lesson: every async function that
might fire a wire call in the action path needs to surface its
source contribution to the caller; the action layer folds them.

### Single load-bearing lesson

**"When a fix-up helper exists, the default is wire it through and
argue exceptions, not skip it and argue the gate."**

mutation surface. M9's create skipped both initially under the

true but didn't cover stale-cache. The wire-it-through default
costs nothing when the helper is already extracted; the explicit-
skip default costs a P1 finding because the reasoning is always
local while the contract is global.

This is symmetric with M8's "always aggregate before acting"
pattern — both are arguments for keeping the established
discipline by default and reasoning explicitly when departing.

**"A new verb's argv shape does not imply a new verb's contract

### Discovery 1 — `ColumnMappingInput` is strictly ID-to-ID

**SDK shape:** `type ColumnMappingInput = { source: ID!, target?: ID }`.
Two slots, both ID-shaped. **No value slot.**

**§3 M11 deliverable:** Accept `--columns-mapping <json>` whose values
are either a target ID string OR `{id, value?}` records, where
`value` re-runs through M5a/M8's column translator on target
metadata. Implies a CLI-side feature (the SDK doesn't surface it
directly).

**Implication:** Supporting value-overrides requires a non-atomic
post-move `change_multiple_column_values` mutation against the
target board. That introduces partial-failure semantics — move
succeeded but the value-override leg failed — with no precedent in
the v0.1+v0.2 envelope shapes. Bulk `item update --where`'s
`applied_count` / `applied_to` decoration is per-item; this would
be cross-leg, a different shape.

**Decision:** Defer value-overrides to v0.3. Ship the simple
`{<src>: <target>}` form (string-to-string) which maps directly to
Monday's `columns_mapping` parameter. The parser rejects the rich
`{id, value?}` form loud with a `usage_error` carrying
`details.hint: "deferred to v0.3 — fire monday item set <iid>
<target>=<value> post-move when you need them"`. Agents
hand-crafting the rich shape see a typed error pointing at the
workaround.

**Rationale.** Atomic single-mutation flow is simpler, matches
Monday's wire shape exactly, and avoids a cross-leg partial-failure
envelope shape. The simple form is enough for the cross-board move
contract — agents who need value translation chain `move` +
`item set` themselves until v0.3 ships the atomic primitive.
v0.3 candidate: a `monday item move --columns-mapping` +
`change_multiple_column_values` two-leg flow with explicit
"two-mutation, partial-success" envelope shape OR Monday adding a
value slot to `ColumnMappingInput` (not signalled).

### Discovery 2 — `move_item_to_board(group_id: ID!)` is mandatory

**SDK shape:** `type MutationMove_Item_To_BoardArgs = { board_id:
ID!, group_id: ID!, item_id: ID!, columns_mapping?: ..., ...}`.
`group_id` is required (not nullable, not InputMaybe).

**cli-design §4.3 line 530 pre-M11:** `monday item move <iid>
--to-group <gid> | --to-board <bid> [--columns-mapping <json>]`
(the `|` indicates mutual exclusion).

**Implication:** The pre-M11 surface form `monday item move <iid>
--to-board <bid>` (no `--to-group`) had no destination on the
target board. Either the CLI had to auto-pick a default group on
the target board (implicit + surprising) or `--to-group` had to be
mandatory for both forms.

**Decision:** Make `--to-group <gid>` required for both same-board
AND cross-board. cli-design §4.3 line 530 updated during the M11
docs sweep: `monday item move <iid> --to-group <gid> [--to-board
<bid>] [--columns-mapping <json>]`. Same-board = `--to-group`
alone; cross-board = `--to-group --to-board`; `--to-board` alone
(no `--to-group`) → `usage_error` ("Monday requires a destination
group").

**The mutex now lives between `--columns-mapping` and `--to-board`'s
absence** — passing `--columns-mapping` without `--to-board` is a
user mistake (mapping is cross-board-only) so the action throws
`usage_error` rather than silently dropping the flag.

**Rationale.** Required-`--to-group` matches Monday's wire shape
without surprising agents with an implicit default. Forces the
agent to think about destination explicitly — same-board moves
need a target group anyway, cross-board moves need both.

### Implementation notes

**Strict default per §8 decision 5.** The unmatched-column check
fires pre-mutation (and pre-emit on dry-run). Every source column
**with actual content** — populated cells per `cellHasData`'s
recursive emptiness check (non-null `value` whose object/array
leaves aren't all empty, OR non-empty `text`) — must either have a
verbatim ID match on target OR be bridged by `--columns-mapping`.
Empty cells (no value, cleared rich shapes like `{}` /
`{personsAndTeams: []}`, or empty `text`) drop out of the check
because Monday wouldn't carry their values across the move anyway.
Unmatched → `usage_error` carrying
`details.unmatched: [{source_col_id, source_title, source_type}]`
+ `details.example_mapping: {<src>: "<target_col_id>"}`. The
example mapping seeds the agent's next call — copy, fill in
target IDs, retry. `--columns-mapping {}` (empty object) is the
explicit "drop everything (Monday's permissive default)" opt-in
that bypasses the check.

**Verbatim ID matches surface explicitly in `column_mappings`.**
Even when source + target IDs match without an agent-supplied
mapping, the planner emits `{source: X, target: X}` in the wire
payload + dry-run echo. Reasoning: the dry-run tells the agent
exactly what Monday will receive on the wire, so a comprehensive
echo is more useful than a partial one. Mapping-overrides win over
verbatim matches (agents can deliberately rename a same-ID
column — e.g., if the target board's `status_4` carries a different
semantics, the agent maps source's `status_4` to target's
`archived_status_4`).

**Inherited helpers (R23, R27, R28, R21).** Both transports use
`projectMutationItem` (R28 — live-mutation null-handling, errorCode:
'not_found'). Same-board dry-run uses `readSourceItemForDryRun`
(R27, operationName: 'ItemMoveRead'). Cross-board uses both R27
+ `lookupItemBoard` (R23) as a board-id fallback for the rare
projection-board-id-null path. `mergeSource` + `mergeCacheAge`
(R21) fold the cross-board path's mixed cache + live legs into the
final envelope source. Net new code stayed small (~530 LOC for
the verb, ~140 for the JSON parser); the inherited helpers paid off.

**Coverage gap.** `move.ts` line 526 (the `lookupItemBoard`
fallback for `sourceItem.board_id === null`) sits at 90% branch
coverage on the file because the fallback fires only when Monday
returns a malformed item with a null board reference — the
projection schema allows it (`board_id: z.string().nullable()`)
but in practice every fixture-driven path returns a non-null
board_id. The branch is genuinely defensive; the file's branch
coverage stays above the 95/94 floor at 90 because the file's
overall surface is small and the one defensive branch dominates
the percentage. Considered + rejected: forcing the test via a
fixture cassette that returns `board: null` on the source-item
read. Rejected because the test would only exercise the fallback
in a synthetic scenario; the production path is the projection's
primary ID with the lookup fallback as a belt-and-braces guard.

```ts
// src/api/source-aggregator.ts (extend existing module)
export class SourceAggregator {
  private source: EnvelopeSource | undefined;
  private cacheAge: number | null = null;

  constructor(
    seed?: { readonly source: EnvelopeSource; readonly cacheAgeSeconds: number | null },
  ) {
    if (seed !== undefined) {
      this.source = seed.source;
      this.cacheAge = seed.cacheAgeSeconds;
    }
  }
  record(source: EnvelopeSource, cacheAgeSeconds: number | null): void {
    this.source = mergeSource(this.source, source);
    this.cacheAge = mergeCacheAge(this.cacheAge, cacheAgeSeconds);
  }
  result(fallback: EnvelopeSource = 'live'): {
    readonly source: EnvelopeSource;
    readonly cacheAgeSeconds: number | null;
  } {
    return { source: this.source ?? fallback, cacheAgeSeconds: this.cacheAge };
  }
}
```

### What pattern explains the findings

The whole chain — round-2 through round-5 — was the same shape:
**a claim about "X round-trips" backed by the wrong evidence.**

- Round 2's claim ("pass already-resolved forms") backed by the
  reasoning "lookup translator accepts the resolved form;
  mutation translator emits the resolved form."
- Round 3's narrowing ("numeric user IDs work") same shape —
  the mutation translator's actual grammar (cli-design §5.3
  step 3) was checked against the help text ONLY at round-3,
  after the help had already shipped one round and made the
  claim.
- Round 4's narrowing ("ISO dates work, other kinds work")
  same shape — the items_page wire shape (the EXACT marker,
  the bare-pipe-string compare for rich grammars) was only
  consulted at round-4, after Codex web-searched Monday's
  changelog.

The reasoning never asked: **is this round-trip pinned by an
existing test, or am I reasoning from the translator's input
shape?** Translator-shape reasoning is one step removed from
wire-shape reality. The wire shape is what Monday's filter
actually compares against, and the only source of truth for
that is either (a) an existing integration test on a sibling
surface (`item list --where`, `item update --where`) that
shipped the same wire shape and pinned the round-trip, or (b)
a new integration test that pins the round-trip explicitly.

### Single load-bearing lesson

> **Wire-shape claims need empirical proof.** When a CLI verb
> sells idempotency / round-trip / "this matches what you just
> wrote," every column-kind + value-form combination on the
> safe list either (a) is pinned by an integration test on this
> verb, or (b) inherits from a sibling surface whose existing
> tests pin the same wire shape. Translator-input-shape
> reasoning ("the lookup translator accepts X; the mutation
> translator emits X") is not enough — the mutation translator
> emits a rich payload, the lookup translator emits a filter
> compare, and Monday's filter compares against the stored rich
> shape. The two paths diverge at the wire layer and only
> empirical evidence proves they reconverge.

Operational rule for M13+: when a milestone makes a "this
round-trips" claim in help / contract, the corresponding
integration test that pins the round-trip MUST be either a new
test in this milestone or an explicitly-cited test from a
sibling surface (with the test path in the post-mortem). No
test cited → the claim doesn't ship in the contract.

### Implementation notes

**Branch decision is a discrete state machine.** `decideBranch`
takes the lookup result's `(matchCount, cursor)` and returns
one of `{ kind: 'create' } | { kind: 'update', itemId } |
{ kind: 'ambiguous', error: ApiError }`, throwing
`ApiError('internal_error', ...)` directly on the empty-with-
cursor anomaly. The three returned states cover the valid
Monday responses (zero matches with null cursor → create, one
match → update, 2+ matches → ambiguous_match wrapped as
`error`); the throw covers the contract anomaly Codex round-1
F3 caught (zero matches with non-null cursor — Monday's
items_page returns `cursor: null` when no more pages exist, so
non-null cursor with empty items is API-side malformation).
Throwing rather than returning a fourth state keeps the union
narrow at the call sites — the create / update / ambiguous
branches are exhaustive over the valid responses.

**The "candidates" in ambiguous_match.** When the lookup
returns 2+ matches, the error envelope's `details.candidates`
carries up to 10 items' `{id, name}` projection so an agent
debugging sees representative items without paying for a
second round-trip. The lookup uses `limit: 11` — 10 items + 1
lookahead lets the planner distinguish "exactly 2..10 matches"
from "11+ matches" in a single page. `details.match_by` /
`details.match_values` / `details.matched_count` round out the
debug surface so an agent can replay the predicate or tighten
it without re-deriving the input shape from argv.

**Bulk clear `--where` reuses bulk `item update --where` almost
verbatim.** The cursor walk, `confirmation_required` gate,
per-item failure decoration, and dry-run shape are inherited
from M5b's bulk update. The only M12-specific code is the
"clear" mutation per-item (one column → null) vs update's
"set" mutation. This is why M12 deliberately bundled both verbs
— the clear extension is mostly free given the upsert
infrastructure was already exercised in test.

**§16 R31 / R32 audit.** R31 (`cellHasData` lift to
`item-projection.ts`) — M12 didn't surface a second consumer.
The M12 lookup leg projects items via the existing
`projectMutationItem` (R28) which doesn't use semantic
emptiness. R31 stays deferred. R32 (JSON-object argv parse
helper) — M12 didn't add a third consumer. The bulk
`--filter-json` parser inherits from `parseFilterJson`
(filters.ts, M4); the `--match-by` parser uses
`splitSetExpression` (R22). R32 stays deferred. Both R31 and
R32 reach their re-evaluation triggers at M14/M15/M16.

- `src/commands/item/create.ts` (M9) — `CREATE_ITEM_MUTATION`,
  `CREATE_SUBITEM_MUTATION` (operation names `ItemCreateTopLevel`
  / `ItemCreateSubitem`).
- `src/commands/item/update.ts` (M5b) —
  `CHANGE_SIMPLE_COLUMN_VALUE_MUTATION`,
  `CHANGE_COLUMN_VALUE_MUTATION`,
  `CHANGE_MULTIPLE_COLUMN_VALUES_MUTATION` (operation names
  `ItemUpdateSetSimple` / `ItemUpdateSet` / `ItemUpdateBulk`).
- `src/commands/item/clear.ts` (M5b + M12) —
  `CHANGE_SIMPLE_COLUMN_VALUE_MUTATION`,
  `CHANGE_COLUMN_VALUE_MUTATION` (operation names
  `ItemClearSimple` / `ItemClear`).
- `src/commands/item/upsert.ts` (M12) — `CREATE_ITEM_MUTATION`,
  `CHANGE_SIMPLE_COLUMN_VALUE_MUTATION`,
  `CHANGE_COLUMN_VALUE_MUTATION`,
  `CHANGE_MULTIPLE_COLUMN_VALUES_MUTATION` (operation names
  `ItemUpsertCreate` / `ItemUpsertSetSimple` / `ItemUpsertSet`
  / `ItemUpsertBulk`).

`CHANGE_SIMPLE_COLUMN_VALUE_MUTATION` and
`CHANGE_COLUMN_VALUE_MUTATION` are now 3-consumer sites
(update + clear + upsert), and `CREATE_ITEM_MUTATION` is a
2-consumer site (create + upsert).
`CHANGE_MULTIPLE_COLUMN_VALUES_MUTATION` is a 2-consumer site
(update + upsert). The mutation strings themselves are short
(~15 LOC each) and identical across sites, but the
**operation_name** differs per command (the SDK's
`request<T>(query, vars, headers)` flow uses the operation
name for telemetry / error attribution).

**Why a candidate.** §16 R30's "lift on third consumer" rule
fires on `CHANGE_SIMPLE_COLUMN_VALUE_MUTATION` and
`CHANGE_COLUMN_VALUE_MUTATION` post-M12. The natural home is
`src/api/item-mutations.ts` exporting the constants + a thin
factory like `createItemMutation(operationName)` that returns
the GraphQL string with the operation name slotted in.

**Why defer.** A clean lift needs a parameterized factory
because each call site uses a different operation name —
either pass the operation name through every call or ship a
factory function per mutation. The factory shape is a small
abstraction tax (one extra import + one factory call per
site) for a savings of ~15 LOC × 4 sites = ~60 LOC — a thin
margin. The duplicate strings also stay byte-identical
because the SDK's `gql` template tag would let any string
drift surface as a GraphQL parse error during integration
tests, so the existing tests already pin the consistency.

**Recommendation: defer until fifth consumer.** M13 is the
update / comment mutation cluster (eight new mutation verbs
mostly thin wrappers around existing `change_*` mutations
the SDK already provides typed calls for); the new verbs
mostly land via the SDK's typed mutations rather than raw
GraphQL strings, so M13 likely won't add a fifth consumer of
the raw `CHANGE_SIMPLE_COLUMN_VALUE_MUTATION` /
`CHANGE_COLUMN_VALUE_MUTATION` constants. M14 / M15
(workspace + board lifecycle) introduce different mutations
entirely. **Trigger phrase:** "I'm copying yet another
`mutation ChangeSimpleColumnValue($itemId: ID! ...)` into a
new commands/* file." Audit at the M14 → M15 boundary.

### Ordering — as deferred

R33 stays deferred to its fifth-consumer trigger. R31 and R32
(post-M11 §16) remain deferred to M14/M15/M16 triggers. No
post-M12 lift work this milestone — M12's net new code stayed
small enough that the existing helper inventory carried the

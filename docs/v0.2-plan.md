# v0.2 implementation plan

> **Status:** plan, not contract. The binding contract is
> [`cli-design.md`](./cli-design.md). This document sequences the work
> needed to ship v0.2 of that contract; it doesn't redefine it. If a
> conflict arises, `cli-design.md` wins and this plan is updated.

## 1. Scope summary

v0.2 is "**the mutating core — agents can drive a backlog**". v0.1
gave agents the read surface plus four narrow mutation verbs (`item
set/clear/update`, `update create`); v0.2 lifts the write surface to
parity with the read surface so an agent can create items, move them
between groups and boards, archive/delete them, post and edit
comment threads, and shape the boards themselves. The friendly
column-value translator gains six more types, plus the long-deferred
`--set-raw` escape hatch.

**In scope (from `cli-design.md` §13 v0.2):**

| Group | Commands |
|-------|----------|
| **column writers** | `--set-raw <col>=<json>` escape hatch + friendly translators for `link`, `email`, `phone`, `tags`†, `board_relation`†, `dependency`† (†tentative — may slip to v0.3 if fixture work surfaces blocking design cost; see §6 risk register) |
| `item` | `create`, `archive`, `delete`, `duplicate`, `move`, `upsert`, bulk `clear --where ... <col>` |
| `update` | `reply`, `edit`, `delete`, `like`, `unlike`, `pin`, `unpin`, `clear-all`; `list --board <bid>`; `list <iid> --with-replies` |
| `workspace` | `create`, `update`, `delete`, `add-users`, `remove-users` |
| `board` | `create`, `update`, `archive`, `delete`, `duplicate`, `add-users` |
| `board` (structure) | `column-create`, `column-update`, `column-delete`, `group-create`, `group-update`, `group-archive`, `group-duplicate`, `group-delete` |
| polish | NDJSON streaming for `item search` and `update list`; refreshed envelope-snapshot suite; `0.2.0` release artefacts |

**Universal partial-success rule (cli-design §6.1 compliant).** Multi-
target verbs (`workspace add-users` / `remove-users`,
`board add-users`, `update clear-all`) emit one **success envelope**
(`ok: true`) with per-target results in `data.results: [...]`. Each
record carries its own `{<target_id_field>, ok: true | false,
error?}` where the field name is verb-specific — `user_id` for
add-users / remove-users, `update_id` for `clear-all`, `<verb>_id`
generally. The shared envelope-builder (`buildPartialSuccessMutation`)
takes the field name as a parameter; the per-record shape is
otherwise uniform (same `ok` and optional `error: { code, message }`).
The envelope's top-level `error` slot is reserved for whole-call
failure (couldn't reach the API, couldn't resolve any target before
the loop began, an unrecoverable validation error). All-failed-but-
each-call-attempted is still `ok: true` — the agent reads
`data.results` to determine outcomes. This avoids a §6.1 contract
revision (Codex round-1 P1).

**Out of scope (deferred per §13 "Explicitly deferred"):**

- `monday dev …` — Monday Dev workflow shortcuts (v0.3).
- `monday auth login` / OAuth profiles / config files / `~/.monday-cli/config.toml` (v0.3).
- `monday status` / `monday usage` diagnostics (v0.3).
- `notification send` / `webhook list/create/delete` (v0.3).
- `item search` cross-board (omit `--board`); `item history`; `board favorites`; `item update --continue-on-error` partial-success bulk (v0.3).
- `item watch`; `--concurrency`; asset upload (`add_file_to_column`); shell completion; `doc list/get`; `team` create/manage (v0.4).
- Boolean filter DSL — v0.2 keeps the v0.1 `--where` operator allowlist + `--filter-json`. The boolean DSL is fixture-driven; only ships when adoption metrics show concrete demand. Pencilled at v0.2 stretch / v0.3 firm.
- `monday item recreate-from-archive` — still no misleading `restore` command (cli-design §5.4 stands).

## 2. Pre-flight: what's already in place

Working at HEAD (`main`, post-`638d49a`):

- **Foundations (M0–M2.5)** — the runner / signal handling / typed errors / redaction / envelope builders / commander wiring / transport seam / retry layer / `MondayClient` / `account` commands all live. v0.2 inherits these unchanged.
- **Read surface (M3–M4)** — `workspace`, `board`, `user`, `update`, `item` reads (incl. filter DSL, cursor pagination, NDJSON streaming for `item list`, projection layer). Every write verb in v0.2 leans on these for resolution: `--board <bid>` lookup, column resolution, `me` token, board metadata cache.
- **Write half of §5.3 (M5a)** — `column-values.ts` (translators for `text`/`long_text`/`numbers`/`status`/`dropdown`/`date`/`people`), `dates.ts`, `people.ts`, `column-types.ts` writable allowlist, `dry-run.ts` engine. **M8's writer expansion plugs into this** — six new translator slots in the same dispatcher, plus the long-deferred `--set-raw` path.
- **Mutation surface (M5b)** — `item set`/`clear`/`update` (single + bulk via `--where`) + `update create`. Shared helpers: `item-helpers.ts`, `resolver-error-fold.ts`, `resolveColumnsAcrossClauses`, `parse-boundary.ts`. **M9–M13 reuse these directly**: bulk `item clear --where` is the per-item leg already plus a cursor walk; `item create`/`upsert`/`move` reuse the resolver fold; `update reply`/`edit`/`delete` reuse `update create`'s body-source plumbing (`--body` / `--body-file` / stdin).
- **Diagnostics + escape hatch (M6)** — `board doctor`, `raw`, agent-flow E2E. `raw` already gives agents a write escape; v0.2's `--set-raw` is the typed alternative for the column-value path specifically.
- **Release scaffolding (M7)** — `published-tarball.test.ts`, `envelope-snapshots.test.ts` (40 snapshots), `output-shapes.md`, `CHANGELOG.md`. **M18's release prep extends each**: ~30 new envelope snapshots, ~30 new `data` shape entries, `0.2.0` changelog cluster.
- **Test infrastructure** — fixture transport, fixture server (E2E), redaction-hardening suite, agent-flow E2E. Coverage floor sits at **94% branches / 95% lines / functions / statements**; v0.2 holds or raises it (never lowers).
- **Tooling** — ESM, NodeNext, strictest TS, ESLint `strictTypeChecked + stylisticTypeChecked`, vitest 4. CI runs typecheck + lint + test:coverage + build smoke on Node 22 and 24. Two-AI review (Codex `gpt-5.5`, read-only) per design change and per implementation milestone.

What's **missing** that v0.2 needs (each filled by the named milestone):

- A **cache-invalidation discipline** for board-metadata-mutating commands (M16). Today, every `--set` consults the cached board metadata; mutations that change board shape (`column-create`/`column-delete`/`group-rename`) need to invalidate the cache so subsequent reads/writes see the new shape. v0.1 had no schema-mutating commands so the question never arose.
- A **partial-success envelope** for multi-target add-users / remove-users (M14, M15). Single-target failure is the v0.1 mutation envelope; for these multi-user ops, per-user `{ok, error?}` records keep the agent informed without abandoning the whole call. Fail-fast bulk update (cli-design §6.5 `details.applied_to`) is the working precedent — `--continue-on-error` for `item update` waits until v0.3.
- A **`--set-raw <col>=<json>`** path through the existing translator (M8). The translator currently dispatches on type; raw bypasses dispatch but still resolves the column ID and validates the JSON shape against zod.
- A **column-mapping JSON shape** for cross-board `item move` (M11). Maps source column IDs to target column IDs with optional value-translation overrides; needs a small zod schema.
- An **idempotency layer** for `item upsert` (M12). Match-by lookup + create-or-update branching; the branch decision is captured in the mutation envelope (`operation: "create_item" | "update_item"`) so agents can reason about which path fired.
- A **board-structure mutation cluster** (M16). Column/group create/update/archive/delete each have one Monday mutation; what's new is the cache invalidation fan-out + the column-type allowlist for `column-create`.

## 3. Milestone sequencing

Eleven milestones (M8–M18; M16 split into columns-then-groups per
Codex P2). Each milestone is a coherent commit cluster (3–10 commits)
ending on a green tree with the listed exit criteria met. Earlier
milestones unblock later ones; do not start M(N) until M(N-1)'s exit
criteria are satisfied. Each milestone updates `docs/output-shapes.md`
for any new command shapes it lands and adds envelope snapshots for
any new emitted shape.

Order at a glance: **writer expansion → item lifecycle → bulk symmetry
+ idempotency → updates → workspace + board CRUD → board columns
(introduces eager invalidation contract) → board groups (reuses it)
→ polish + release**.

```
       M8                                 — writer expansion (foundations)
       M9 ─ M10 ─ M11                     — item lifecycle (create / archive+delete+duplicate / move)
       M12                                — item upsert + bulk item clear
       M13                                — update mutations + update list expansions
       M14 ─ M15                          — workspace + board lifecycle
       M16                                — board columns (cache-invalidating; also pins the eager-invalidation contract)
       M17                                — board groups (cache-invalidating, smaller surface)
       M18                                — NDJSON streaming + 0.2.0 release prep
```

### M8 — Writer expansion + `--set-raw` escape hatch

The foundational v0.2 milestone. No new commands ship — instead,
the column-value translator gains six new type slots and the
`--set-raw` escape hatch, so every later milestone (M9 `item
create`, M11 `item move --columns-mapping`, M12 `item upsert`,
M16 `column-create`) can lean on the full writable type surface
without re-touching tests. **Foundations-first** is the load-bearing
ordering decision (§8 decision 1).

**Deliverables**
- `src/api/column-values.ts` — six new translator slots:
  - `link` — `<url>` (one segment) → `{url:<url>, text:<url>}`; `<url>|<text>` (pipe-split, max 1 split, both segments trimmed) → `{url, text}`. `text` defaults to the URL when the pipe form is absent. URL validated via `z.string().url()`; failure → `usage_error`.
  - `email` — single email → `{email:<value>, text:<value>}`; `<email>|<text>` → `{email, text}`. Email validated via `z.string().email()`; failure → `usage_error`.
  - `phone` — `<phone>|<country>` (pipe form mandatory) → `{phone:<phone>, countryShortName:<country>}` where `<country>` is a 2-letter ISO 3166-1 alpha-2 code (uppercase). E.164-loose validation (`+?\d{6,15}`); ISO code validated against a frozen allowlist (~250 codes from `Intl` / a vendored list). Single-segment `<phone>` (no `|<country>`) rejected with `usage_error` — Monday's phone-column validation requires both number and country code AND verifies they match (per Monday's phone-validation changelog), so the friendly translator can't safely default `countryShortName: ""`. Agents who need empty-country phones use `--set-raw`.
  - `tags` (tentative) — comma-split tag names → `{tag_ids:[N1,N2]}` via account-tag lookup. Adds `tag-directory.ts` (mirrors `user-directory` from M5a). New error code `tag_not_found` (27th stable code if `tags` ships firm; see §8 decision 2). Slips to v0.3 if the per-account `tags` query proves too expensive to cache (fixture work in M8 close decides).
  - `board_relation` (tentative) — comma-split item IDs → `{item_ids:[N1,N2]}`. Item IDs are accepted directly (no name-resolution sugar in v0.2 — `item find <name>` first, then `--set`). Cross-board references validated against the source column's allowed-boards settings — Monday exposes `boardIds` (array) or `boardId` (singular) per `get_column_type_schema`; the CLI normalises to `allowed_boards = settings.boardIds ?? [settings.boardId]`. Off-board → `usage_error` with `details.allowed_boards`. Slips to v0.3 if linked-board enumeration proves to require per-call complexity-budget design.
  - `dependency` (tentative) — same shape as `board_relation` but uses Monday's separate `dependency` payload. Same slip risk.
- `src/api/raw-write.ts` — the `--set-raw <col>=<json>` path. Resolves `<col>` through the same `resolveColumnWithRefresh` flow as `--set`. Two reject lists fire after resolution but before mutation (matching cli-design §5.3 escape-hatch contract):
  - **Read-only-forever** (`mirror`, `formula`, `auto_number`, `creation_log`, `last_updated`, `item_id`) → `unsupported_column_type` with `read_only: true`.
  - **`files`-shaped** (`file`, anything else where Monday's write path is `add_file_to_column` rather than `change_column_value`) → `unsupported_column_type` with `deferred_to: "v0.4"`.
  Otherwise validates `<json>` parses as a `JsonObject`, then constructs a `TranslatedColumnValue` with `payload: { format: 'rich', value: <parsed> }` so the existing `selectMutation` dispatcher handles it uniformly. **No type-shape validation** beyond the JSON-object boundary — the user took the escape hatch and owns wire-shape correctness; Monday's server-side rejection surfaces as `validation_failed`. Mutually exclusive with `--set` for the same `<col>` (resolution-time enforced — the duplicate-ID check fires after both flags' tokens resolve to column IDs, since argv-parse can't tell whether two distinct tokens alias to the same column without board metadata; cli-design round-2 P2 correction).
- `src/api/column-types.ts` — `WRITABLE_COLUMN_TYPES` grows from 7 to 13 (10 firm, 3 tentative). The frozen `as const` array's order is part of the contract; tests iterate it. Update `commands/board/describe.ts` so `example_set` covers the new types.
- `unsupported_column_type` — for v0.1's `link`/`email`/`phone`/`tags`/`board_relation`/`dependency` row, `deferred_to: "v0.2"` → either disappears (type now writable) or, for the tentative slips, becomes `deferred_to: "v0.3"`. Read-only-forever and `deferred_to: "future"` rows unchanged.
- `monday item set` and `monday item update` learn `--set-raw <col>=<json>`. Bulk `item update --where` learns `--set-raw` too. `--set` and `--set-raw` against the same column are mutually exclusive (resolution-time enforced; `usage_error`). Help text gets an example block.
- `cli-design.md` §5.3 step 4 + writer-expansion roadmap table + §4.3 command-tree (adds `--set-raw` to the relevant signatures) updated to reflect the v0.2 firm rows. M8 ships its own design diff for the design doc — submitted alongside the M8 PR for two-AI review per CLAUDE.md.

**Tests (pyramid)**
- Unit: each new translator's grammar — happy path, every error path, Unicode in the human-readable text segment, pipe-split edge cases (empty trailer, multiple pipes, leading/trailing whitespace), idempotent translation (re-running translation on the output reproduces the wire shape). Wire-shape fixture pinned per (count × type) cell so v0.3 inherits the rule unchanged. `--set-raw` coverage: column resolution succeeds, JSON parse fails (`usage_error`), JSON parse succeeds and round-trips through `selectMutation`, `--set` and `--set-raw` against the same column collide (`usage_error`).
- Integration: `item set --board <bid> <link-col>=https://example.com|Site` produces the expected wire payload (cassette pinned). Repeat for each new type. `--set-raw` against an `unsupported` / `future` column type that accepts `change_column_value` *succeeds* at the CLI layer (the escape hatch's whole point) and surfaces Monday's rejection as `validation_failed`. `--set-raw` against read-only-forever (`mirror` / `formula` / etc.) or `files`-shaped (`file`) → `unsupported_column_type` at the CLI layer, no API call fires.
- E2E: one happy-path spawn per new firm type (`link`, `email`, `phone`); the tentative types ship with E2E too if they make the cut.

**Exit**
- `monday item set <iid> --board <bid> link_col=https://x` succeeds against fixtures.
- `monday item set <iid> --board <bid> --set-raw status='{"label":"Done"}'` succeeds against fixtures and equals the friendly `--set status=Done` payload byte-for-byte.
- `unsupported_column_type` `deferred_to: "v0.2"` no longer fires for any of the firm v0.2 row types.
- `board describe`'s `example_set` covers all newly-writable types.
- Two-AI review pass cleared.
- Coverage floor held at 94/95/95/95.

### M9 — `item create` (incl. subitem creation + position)

The first net-new item-mutation verb. Non-idempotent (creating an
item with the same name twice yields two items — cli-design §5.8);
ships with `--dry-run`.

**Deliverables**
- `src/commands/item/create.ts` — `monday item create --board <bid> --name <n> [--group <gid>] [--set <col>=<val>]... [--set-raw <col>=<json>]... [--parent <iid>] [--position before|after --relative-to <iid>] [--dry-run]`. Calls `create_item` (or `create_subitem` when `--parent` is present) with the full `--set` payload bundled into `column_values`. **Single round-trip is a hard exit gate** (Codex P1): if Monday's `create_item.column_values` proves not to accept any v0.2 type's wire shape, M9 halts; we don't fall back to `create_item` + `change_multiple_column_values` because a partial-state failure between the two calls leaves an item with the requested name but missing column values. The v0.1 fixture pass against `column_values` covers all seven v0.1 types; M8's six new types each get a fixture-pinned wire-shape test on the create path before M9 implementation begins. `--position` resolves to Monday's `position_relative_method` + `relative_to`.
- `--parent <iid>` triggers `create_subitem`. Subitems live on Monday's auto-generated subitems board; column resolution targets *that* board, not the parent's. M9 plumbs the resolved subitem-board ID through the same column-resolution path as the top-level path.
- `--position before|after --relative-to <iid>` — both flags required together; one without the other is `usage_error`. `relative_to` must reference an item on the same board (verified by item lookup, mirrors M5b's wrong-board check).
- `idempotent: false` in the CommandModule. Help text calls out `item upsert` (M12) for the idempotent variant.
- Mutation envelope: `data: { id: <new_item_id>, name, board_id, group_id, parent_id? }` plus the v0.1 top-level `resolved_ids: { ... }` slot per cli-design §6.4 (Codex round-2 P2: v0.1 mutation contract uses top-level `resolved_ids`, not nested in `data`). Snapshot pinned.
- Resolver-warning fold: every `--set`-resolution warning that surfaces during the create flow folds into `details.resolver_warnings` on failure (mirrors M5b discipline).

**Tests (pyramid)**
- Unit: argv parser — `--position` requires `--relative-to`; `--parent` and `--group` are mutually exclusive (subitems live on the subitems board, not in groups); `--name` empty after trim → `usage_error`; multiple `--set` against the same column → `usage_error`.
- Integration: happy path (top-level item, default group, multiple `--set`); subitem path (`--parent <iid>`); position path (before / after); each error path (`not_found` on board, `not_found` on parent, `not_found` on relative-to, `validation_failed` on Monday rejection); dry-run path **for both branches** — top-level `operation: "create_item"` AND subitem `operation: "create_subitem"` envelopes pinned (Codex round-4 P2 — both planned-change operations are §9 preconditions; tests must assert both shapes).
- E2E: one happy-path spawn (top-level), one subitem spawn.

**Exit**
- `monday item create --board <bid> --name "Test" --set status=Done` returns a new item ID against fixtures.
- Subitem path verified (top-level create + subitem create round-trip).
- `--dry-run` echoes the planned create without firing the mutation.

### M10 — `item archive` + `item delete` + `item duplicate`

Lifecycle siblings. Share the confirmation-gate pattern from M5b's
bulk paths (`--yes` required for destructive verbs).

**Deliverables**
- `src/commands/item/archive.ts` — `monday item archive <iid> --yes [--dry-run]`. Calls `archive_item`. `--yes` required for live archive (destructive — archived items hide from default queries and Monday only retains them for a 30-day recovery window). cli-design §4.3 line 509 currently shows `item archive <iid>` without `--yes`; §9 precondition adds the `--yes` to the command-tree before M10 ships. `--dry-run` reports the would-archive item state. Idempotent on the wire (re-archiving an archived item is a no-op on Monday's side); CLI marks `idempotent: true`.
- `src/commands/item/delete.ts` — `monday item delete <iid> --yes [--dry-run]`. Calls `delete_item`. `--yes` mandatory (truly destructive — Monday's delete is recoverable for 30 days but the CLI doesn't promise that). Not idempotent (re-deleting an already-deleted item is `not_found`).
- `src/commands/item/duplicate.ts` — `monday item duplicate <iid> [--with-updates] [--dry-run]`. Calls `duplicate_item` with `with_updates` boolean. Returns the new item's ID + the source item's ID. **Not idempotent** — every call creates a new item.
- All three reuse `runByIdLookup` (M4 R7) for the source-item read. All three echo `meta.source` correctly (cache + live for the lookup leg).

**Tests (pyramid)**
- Unit: `--yes` enforcement on `archive`/`delete`; `--with-updates` boolean parser; idempotency-flag in CommandModule shape.
- Integration: each verb happy path; each error path (`not_found`, `forbidden`, `validation_failed`); `confirmation_required` without `--yes` for `delete`; dry-run for each.
- E2E: one happy-path spawn for `delete --yes` (it's the highest-blast-radius verb); the tarball test learns the new commands.

**Exit**
- All three verbs return correct envelopes against fixtures.
- `monday item delete <iid>` without `--yes` exits 1 with `confirmation_required`.

### M11 — `item move` (group + cross-board)

The most design-loaded item-lifecycle verb. Cross-board moves
re-resolve columns on the target and may need a `--columns-mapping
<json>` to translate values that don't share IDs.

**Deliverables**
- `src/commands/item/move.ts` — `monday item move <iid> (--to-group <gid> | --to-board <bid>) [--columns-mapping <json>] [--dry-run]`. Mutually exclusive `--to-group` / `--to-board`.
  - **Same-board (group move):** calls `move_item_to_group`. No column re-resolution needed.
  - **Cross-board move:** calls `move_item_to_board`. Re-resolves every column on the source item against the target board's metadata. `--columns-mapping <json>` accepts `{<source_col_id>: <target_col_id> | { id: <target_col_id>, value: <translation override> }}`. Without mapping: source column IDs must match target column IDs verbatim (Monday's default behaviour); unmatched columns surface as `usage_error` with a `details.unmatched: [{source_col_id, ...}]` list and a `--columns-mapping` example.
- `src/api/column-mapping.ts` — zod schema for `--columns-mapping`. `JsonObject` with values being either a target column ID (string) or `{id, value?}` records. `value` overrides take a column-translator-friendly value that re-runs through M5a/M8's translator on the target board's column metadata.
- Cross-board moves preserve item updates (Monday's behaviour); CLI documents this in `--help`.
- `--dry-run` reports planned column mappings + any unmatched columns (`usage_error` still raised so the agent doesn't have to interpret a "would-fail" dry-run shape).

**Tests (pyramid)**
- Unit: mapping-JSON parse boundary (R18); same-board vs cross-board branch; `--to-group` / `--to-board` mutual exclusion; unmatched-column error decoration.
- Integration: same-board happy path; cross-board happy path with explicit mapping; cross-board with no mapping and matching column IDs (succeeds); cross-board with no mapping and divergent IDs (`usage_error`); cross-board with mapping override that re-runs translation.
- E2E: one cross-board happy path with mapping.

**Exit**
- Same-board and cross-board paths both verified against fixtures.
- `--columns-mapping` JSON parses and round-trips through translation.
- `--dry-run` reports the mapping plan.

### M12 — `item upsert` + bulk `item clear --where`

Idempotency cluster. Both verbs deliberately ship together: `item
upsert`'s match-then-update branch reuses M5b's `item update` logic
verbatim, and bulk `item clear --where` reuses M5b's bulk `item
update --where` cursor walk. Two coherent additions in one
milestone.

**Deliverables**
- `src/commands/item/upsert.ts` — `monday item upsert --board <bid> --name <n> --match-by <col>[,<col>...] [--set <col>=<val>]... [--set-raw <col>=<json>]... [--dry-run]`. **Sequential-retry idempotent** (cli-design §5.8) — re-running with the same args from the same agent is safe; concurrent agents are *not* a uniqueness guarantee (Codex round-1 P1: two agents observing zero matches at the same instant both create). Match-by lookup → if 0 matches, branch to `create_item` (M9 surface reused); if 1 match, branch to the v0.1 `item update` logic (M5b's existing single-item path: synthetic `name` key bundled into `change_multiple_column_values` when `--name` accompanies `--set`, or `change_simple_column_value(column_id: "name")` when `--name` is the only diff — Codex round-4 P2 alignment with the v0.1 contract; M12 reuses `commands/item/update.ts`'s entry point rather than re-implementing rename); if 2+ matches, fail with `ambiguous_match` (28th stable error code if both v0.2 codes ship, 27th if `tags` slips; see §8 decision 2). The mutation envelope's `data.operation: "create_item" | "update_item"` exposes the branch (`data.operation` not `meta.operation` — Codex round-2 P2; `meta` has no `operation` slot, and v0.1's mutation envelope already keeps operation-shape signals in `data`). Race mitigation guidance lives in `--help`: agents should use a stable hidden key column for `--match-by` and tighten the predicate so a duplicate from a race surfaces as `ambiguous_match` on the next call. Concurrent-write protection is a v0.4 candidate (cli-design §9.3 lock-resource semantics).
- `--match-by` accepts column tokens (resolved via `resolveColumnsAcrossClauses`) plus the literal `name` token, which matches against item name. Multiple match-by columns AND-combine (item must equal all named values). Match-by values come from the corresponding `--set` flag's value (which is required for every match-by column).
- `src/commands/item/clear.ts` — bulk `monday item clear --board <bid> <col> --where <c>=<v>... | --filter-json <json> --yes [--dry-run]`. Per-item leg already shipped (M5b's `translateColumnClear`). Cursor walk + `confirmation_required` + per-item failure decoration mirror bulk `item update --where`. Bulk dry-run aggregates per-item `planChanges` results into one N-element `planned_changes` array, dedupes warnings.
- `cli-design.md` §6.5 grows from 26 codes to 28 if both `tag_not_found` (M8, conditional on `tags` shipping firm) and `ambiguous_match` (M12) land; 27 if `tags` slips. §13's command tree updated.

**Tests (pyramid)**
- Unit: upsert match-by parser (multi-column AND); branch-decision logic given 0 / 1 / 2+ match-row counts; bulk-clear parser reuses bulk-update test scaffolding.
- Integration: upsert happy path (create branch + update branch); upsert ambiguous-match (`ambiguous_match`); bulk clear happy path with `--where`; bulk clear `confirmation_required` without `--yes`; bulk clear per-item failure decoration (matching the §6.5 `applied_to` shape).
- E2E: one upsert happy path (each branch); one bulk-clear happy path.

**Exit**

## 4. Cross-cutting "definition of done" per command

v0.2 inherits v0.1's §4 checklist verbatim with four additions:

12. **Cache invalidation discipline.** Any command that mutates
    board metadata — column or group shape (M16, M17) — calls
    `invalidateBoard(boardId)` after a successful response. The
    v0.1 board-metadata cache includes both `columns: [...]` and
    `groups: [...]`; either kind of mutation invalidates. Documented
    in the command's source header. The integration test exercises
    a `mutate → read → see fresh state` round-trip AND asserts the
    cache-miss-refresh path did not fire (eager invalidation, not
    lazy refresh).
13. **Partial-success envelope shape pinned.** Multi-target commands
    that emit per-target results (`workspace add-users` /
    `remove-users`, `board add-users`, `update clear-all`) document
    the `data.results: [...]` slot in `output-shapes.md` and pin
    one envelope snapshot per verb. Envelope is always `ok: true`
    when dispatch ran (per §1 universal rule); top-level `error`
    only on whole-call failure.
14. **Every mutation verb supports `--dry-run`.** cli-design §9.2
    is binding — there are no exceptions. Verbs whose underlying
    Monday call isn't itself a column-value mutation (e.g.
    `archive_item`, `delete_workspace`, `like_update`) get a
    minimal dry-run shape: `operation: "<mutation_name>"` plus the
    primary IDs, no diff section. cli-design §6.4 receives a
    design diff per milestone listing the new `planned_changes.
    operation` values (§9 preconditions).
15. **Idempotency reasoning written down.** Every M9–M17 command's
    source header has a one-line idempotency rationale (yes / no /
    "yes when X branch fires") so a future agent reading the file
    doesn't have to reverse-engineer it from cli-design.md.

## 8. Decisions (open questions to close before M8)

These questions are open at v0.2-plan draft time. Each gets a decision
recorded here before M8 starts so implementation doesn't relitigate.
The two-AI review on this plan should land each one.

1. **Foundations-first vs agent-value-first ordering.** Closed: foundations-first (M8 before any new mutation). Re-touching M9–M17 when M8 lands later is roughly 5 milestones × test-suite extension cost; foundations-first eats one milestone of "no visible progress" but saves the rework. The visible-progress alternative (M9 first, M8 inserted later) is rejected.
2. **Stable error code count grows from 26 to 27 or 28.** Closed: `tag_not_found` (M8, only if `tags` ships firm — adds 1 to v0.1's 26), `ambiguous_match` (M12 — adds 1 unconditionally). Total: 27 if `tags` slips, 28 if both ship. A 29th slot reserved for a possible `cross_board_columns_unmatched` if M11's `usage_error` decoration proves insufficient — pencilled, not committed.
3. **Partial-success envelope shape — `data.results: [...]` vs new top-level `results`.** Closed: `data.results: [...]` reuses the existing mutation envelope. **The envelope is always `ok: true` when dispatch ran**, even when every per-target call inside the loop failed — the call's contract is "I ran the dispatch and here are the per-target outcomes." Top-level `error` (`ok: false`) is reserved for whole-call failure (couldn't reach API, couldn't resolve any `--users` token before the loop began, etc.). This avoids a §6.1 contract revision (Codex round-1 P1: my draft incorrectly said `ok: false` when all targets fail). Per-target record shape: `{<verb_target_id>: string, ok: true | false, error?: { code, message }}` where the id field is `user_id` for add-users / remove-users and `update_id` for `clear-all`. The shared `buildPartialSuccessMutation` helper accepts the field name as a parameter so the per-verb shape is self-documenting in JSON without forcing every consumer to adopt a generic `target_id`.
4. **`item upsert --match-by` accepts `name` or only column tokens?** Closed: accepts `name` as a special token (matches `item.name`). Justification: `name` is the most common match-by case for "is this item already there?" and forcing agents to first map name to a column ID feels wrong. Documented in cli-design §5.8 update.
5. **Cross-board `item move` default — strict (require mapping) or permissive (drop unmatched)?** Closed: strict. Monday's permissive default silently drops column values; the CLI rejects with `usage_error` and a `--columns-mapping` example. `--columns-mapping {}` (empty object) is the explicit "drop everything" opt-in.
6. **Cache invalidation — eager or lazy?** Closed: eager. Each M16 column-mutation calls `invalidateBoard(boardId)` post-success. The cache-miss-refresh path (cli-design §8) stays as a backstop for cache-state drift between processes (one CLI invocation invalidates the cache; a parallel CLI invocation in another process sees the stale cache until its own call). Cross-process invalidation is a v0.3 candidate. Requires a cli-design §8 design diff (§9 precondition).
7. **`column-create --type` allowlist enforcement.** Closed: validate against known Monday column types (any of the ~40-type set), warn (don't block) when outside `WRITABLE_COLUMN_TYPES`. Warning is **category-branched** per cli-design §5.3 escape-hatch contract — raw-writable types suggest `--set-raw`; read-only-forever flag that there's no write path at all; `files`-shaped types flag the v0.4 deferral. Blocking would be paternalistic — agents may legitimately create read-only columns.
8. **`update clear-all` — sequential delete or parallel?** Closed: sequential (mirrors M5b bulk update). Parallel waits for the v0.4 `--concurrency` flag.
9. **Archive `--yes` consistency across nouns.** Closed: every archive verb requires `--yes`. cli-design §3.1 says destructive ops without `--yes` return `confirmation_required`; archive hides items/boards/groups and only Monday's UI can recover them within ~30 days, so it counts as destructive. cli-design §4.3 line 509 currently shows `item archive` without `--yes` — §9 precondition adds it. Same for `group-archive`. Codex P1 caught the inconsistency in my v0.2-plan first draft.
10. **M16 split — columns first, then groups.** Closed: M16 ships columns (`column-create/update/delete`) and **introduces the eager-invalidation contract** + `invalidateBoard(boardId)` helper. M17 ships groups (`group-create/update/archive/duplicate/delete`) and **reuses** the M16 invalidation contract — both column and group mutations invalidate the v0.1 board-metadata cache (the cache holds both `columns: [...]` and `groups: [...]`; an earlier draft of this decision claimed groups had no cache pressure — Codex round-2 P1 caught the misclaim). M18 absorbs what was the old M17 (NDJSON + release prep). Justification: cache invalidation is the highest-risk design surface in board-structure mutations; concentrating the *contract introduction* in M16 with three verbs keeps the round-trip tests reviewable. Codex round-1 P2 caught the over-stuffed M16 in the first draft.
11. **Every mutation supports `--dry-run` — no exceptions.** Closed: cli-design §9.2 is binding. Verbs whose underlying Monday call isn't a column-value mutation (e.g. `archive_item`, `delete_workspace`, `like_update`) get a minimal dry-run shape: `operation: "<mutation_name>"` plus the primary IDs, no diff section. cli-design §6.4 receives a per-milestone design diff listing the new `planned_changes.operation` values (§9 preconditions). Codex P1 caught the omission in my first draft.

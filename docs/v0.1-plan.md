# v0.1 implementation plan

> **Status:** plan, not contract. The binding contract is
> [`cli-design.md`](./cli-design.md). This document sequences the work
> needed to ship v0.1 of that contract; it doesn't redefine it. If a
> conflict arises, `cli-design.md` wins and this plan is updated.

## 1. Scope summary

v0.1 is "**the read-only core + safe mutations**" — an agent can read
everything the CLI surfaces, make small scoped idempotent changes,
and post comments narrating its work.

**In scope (from `cli-design.md` §13):**

| Group | Commands |
|-------|----------|
| `account` | `whoami`, `info`, `version`, `complexity` |
| `workspace` | `list`, `get`, `folders` |
| `board` | `list`, `get`, `find`, `describe`, `doctor`, `subscribers`, `columns`, `groups` |
| `item` | `list`, `get`, `find`, `search`, `set`, `clear`, `update`, `subitems` |
| `update` | `list`, `get`, `create` |
| `user` | `list`, `get`, `me` |
| `cache` | `list`, `clear`, `stats` |
| `config` | `show`, `path` |
| `schema` | full + per-command (JSON Schema 2020-12) |
| `raw` | positional `<query>`, `--query-file`, `--vars`, `--vars-file` |
| globals | every flag in §4.4, every error code in §6.5 (26 codes), exit codes 0/1/2/3/130 |

**Out of scope (deferred per §13 "Explicitly deferred"):**

- Any `create`/`move`/`archive`/`delete`/`duplicate`/`upsert` on
  items, boards, columns, groups, workspaces (v0.2).
- `update reply/edit/delete/like/pin/clear-all` (v0.2).
- Boolean filter DSL — only `--where` (operator allowlist) +
  `--filter-json` in v0.1 (v0.2).
- Broad column-type writes — v0.1 allowlist is **only** `status`,
  `text`, `long_text`, `numbers`, `dropdown`, `date`, `people`. All
  other types return `unsupported_column_type` with per-category
  guidance: v0.2-roadmap types (`link` / `email` / `phone` /
  `tags` / `board_relation` / `dependency`) carry
  `deferred_to: "v0.2"`; read-only-forever types (`mirror` /
  `formula` / `auto_number` / `creation_log` / `last_updated` /
  `item_id`) carry `read_only: true` with a source-column hint;
  other types carry `deferred_to: "future"`. The `--set-raw`
  escape hatch is itself deferred to v0.2's writer-expansion
  milestone (M5b post-mortem Path B); v0.1 has no raw-write
  surface.
- `monday dev …` (v0.3), `monday auth login` / OAuth profiles (v0.3),
  config files (v0.2 stretch / v0.3 firm).
- `monday item watch`, `--concurrency`, asset uploads, shell
  completion (v0.4).
- `monday item recreate-from-archive` (TBD; explicitly **not**
  shipping a misleading `restore`).

## 2. Pre-flight: what's already in place

Working at HEAD (`master`):

- `src/config/load.ts` — env-based config (token + version + URL +
  timeout) with a strict zod schema. Branch-comprehensive unit
  tests in `tests/unit/config.test.ts`. **Bug discovered during
  review:** does not call `dotenv.config()` despite docs saying
  `.env` is loaded. Fix in M0.
- `src/cli/index.ts` — bare commander entry. Wires `--version` and
  `--help`. Catch-all logs to stderr, exits `1`. **Will need
  rework** before v0.1 ships: it doesn't emit the §6 envelope and
  uses exit `1` for everything (should be `3` for config, `2` for
  API, etc.). Also coverage-excluded today — M0 splits the
  testable runner into `src/cli/run.ts` so coverage applies.
- `src/index.ts` — library re-exports (`loadConfig`, `Config`).
- CI runs `typecheck + lint + test:coverage` on Node 22 and 24,
  plus a `--version` / `--help` smoke-test against the compiled
  binary. Coverage threshold: 90% lines/branches/functions/
  statements.
- Tooling: ESM, NodeNext, strictest TS, ESLint
  `strictTypeChecked + stylisticTypeChecked`, vitest 4.

What's **missing** that v0.1 needs:

- `src/api/`, `src/commands/`, `src/utils/`, `src/types/`
  directories (skeleton folders, no files).
- Output-formatting layer (envelope, JSON/table/text/ndjson
  renderers, truncation logic).
- API client wrapper around `@mondaydotcomorg/api`, the error
  mapper, retry/backoff, complexity logging, raw-GraphQL escape.
- Local cache (boards, users) at `$XDG_CACHE_HOME/monday-cli/`.
- Branded ID schemas, request-ID generator, redaction helper.
- A runtime dependency for table rendering (`cli-table3` —
  decision closed in §8) and a UUID source (`crypto.randomUUID`
  from stdlib).
- Integration / E2E test scaffolding (`tests/integration/`,
  `tests/e2e/`, `tests/fixtures/load.ts`, plus a tiny in-process
  HTTP fixture server for E2E — see §5.3).
- A `Transport` interface (`src/api/transport.ts`) so tests can
  swap out `fetch` without bypassing header injection / retry
  layers (see §5.2).
- Executable zod output schemas per command — JSON Schema
  emission via `z.toJSONSchema` (zod v4 native) rather than
  separate static descriptors (decision closed in §8).

## 3. Milestone sequencing

Ten milestones (M5 split per Codex review §3, M2.5 inserted
post-M2 — see "Pre-M3 refactor pass" below). Each
milestone is a coherent commit cluster (3–10 commits) ending on a
green tree with the listed exit criteria met. Earlier milestones
unblock later ones; do not start M(N) until M(N-1)'s exit criteria
are satisfied. Each milestone updates `docs/output-shapes.md` for
any new command shapes it lands.

Order at a glance: foundations → local commands → API client +
account → **pre-M3 refactor pass** → broad reads + `board
describe` core + column read-resolver → item reads + filtering +
pagination → column writers + dry-run → item mutations +
`update create` →
`board doctor` + `raw` + agent-flow E2E → release prep.

### M0 — Foundations & utilities

Pure infrastructure. No new commands, but everything below
imports from here.

**Deliverables**
- Fix `src/config/load.ts` — actually call `dotenv.config()` so
  `.env` loading matches the documented behaviour. Add a unit
  test that confirms `.env` values are read and that
  `process.env` overrides them.
- `src/cli/run.ts` — testable CLI runner: `run({ argv, env,
  stdout, stderr, stdin, isTTY, clock, transport })`. Catch-all
  converts any error into the §6 envelope and returns the right
  exit code (1 / 2 / 3 / 130). Signal handlers (`SIGINT` →
  abort + flush + 130). Coverage applies here.
- `src/cli/index.ts` shrinks to a 5-line shebang entrypoint that
  calls `run({ argv: process.argv, env: process.env, ... })`.
  Stays coverage-excluded.
- `src/utils/errors.ts` — `ConfigError`, `ApiError`, `UsageError`,
  `CacheError`, all inheriting a base with a typed `code` from the
  v0.1 stable error code set (26 codes — §6.5). Each error carries
  `code`, `message`, `httpStatus?`, `mondayCode?`, `requestId?`,
  `retryable`, `retryAfterSeconds?`, `details?`. Distinguishes
  abort *reason*: `timeout` vs `SIGINT` vs explicit cancel — both
  must not collapse onto exit 130.
- `src/utils/redact.ts` — deep-clone-and-redact helper with a
  configurable known-sensitive key list (`apiToken`,
  `Authorization`, `MONDAY_API_TOKEN`, plus a generic
  `*token*`/`*secret*` regex).
- `src/utils/request-id.ts` — `crypto.randomUUID()` wrapper for
  deterministic injection in tests.
- `src/utils/logger.ts` — stderr-only structured logger.
  Levels: `error`, `warn`, `info` (TTY only), `debug` (verbose
  only). Pipes every payload through `redact()` before write.
- `src/utils/output/envelope.ts` — envelope builders:
  - `buildSuccess<T>(data, meta, warnings?)` →
    `{ ok: true, data, meta, warnings }`
  - `buildError(error, meta)` →
    `{ ok: false, error, meta }`
  - `buildDryRun(plannedChanges, meta, warnings?)` →
    `{ ok: true, data: null, meta: { ...meta, dry_run: true },
       planned_changes, warnings }`
  - `buildMutation<T>(data, meta, sideEffects?, warnings?)` adds
    `side_effects` per §6.4.
  - Stable field ordering for snapshot tests.
- `src/utils/output/select.ts` — picks renderer:
  honours `--json` / `--table` / `--output <fmt>`,
  `MONDAY_OUTPUT`, `process.stdout.isTTY`. Returns `'json' |
  'table' | 'text' | 'ndjson'`. Mutually exclusive flag pairs
  raise `usage_error`.
- `src/utils/output/json.ts`, `…/table.ts`, `…/text.ts`,
  `…/ndjson.ts` — renderers. `table` honours `--full`,
  `--width`, `--columns`. JSON / NDJSON never truncate. NDJSON
  collections emit one resource per line; trailer is
  `{"_meta":{...}}` (§6.3).
- `src/types/ids.ts` — branded zod schemas: `BoardIdSchema`,
  `ItemIdSchema`, `ColumnIdSchema`, `GroupIdSchema`, `UserIdSchema`,
  `WorkspaceIdSchema`, `UpdateIdSchema`. Plus `inferred` types.
- `src/types/global-flags.ts` — global-flag zod schema; one
  source of truth shared by every command. Wires runtime
  behaviour for **every** §4.4 flag including `--minimal`,
  `--quiet`, `--verbose`, `--no-color`, `--no-cache`,
  `--timeout`, `--retry`, `--yes`, `--profile`. Each flag's
  effect has a unit test.
- `src/api/transport.ts` — `Transport` interface
  (`request(query, variables, headers, signal): Response`)
  with a `FetchTransport` implementation that owns header
  injection (`Authorization`, `API-Version`) and timeout. The
  `Transport` instance is injected through `run({ transport })`
  so tests swap in fixtures without bypassing headers/retry.
- Per-command exports use **executable zod output schemas**
  (`outputSchema: z.ZodType`), not separate static descriptors.
  `monday schema` (M1+) emits via `z.toJSONSchema(schema)`
  (zod v4 native). Tests validate emitted command output
  against `outputSchema` to catch drift.

**Tests (pyramid)**
- Unit (the bulk): every error class instantiates with the right
  shape; `redact()` strips known-sensitive keys recursively and
  on arrays / nested objects; envelope builders produce stable
  field order; renderer selection covers TTY/non-TTY/explicit-flag
  matrix; table truncation hits the floor (12 chars) and
  ellipsis; NDJSON trailer well-formed.
- No integration / E2E in M0 (no commands yet).

**Exit**
- Coverage ≥ 90% on `src/utils/**` and `src/types/**`.
- `monday --version` / `monday --help` still pass smoke-test.
- A deliberately thrown `ConfigError` from inside `loadConfig`
  produces the §6 error envelope on stderr and exits `3`.

### M1 — Local-only commands (`config`, `schema`, `cache`)

The smallest user-visible surface. None of these touch Monday's
API, which makes them perfect for shaking out the envelope, the
schema introspection, and the cache layer before the network
joins the picture.

**Deliverables**
- `src/api/cache.ts` — XDG-aware path resolution, 5-minute TTL,
  mode-`0600` writes, `fs.fstat`-on-read mode check, atomic
  rename for safe replacement. Used in M3+ for board metadata
  and the user directory.
- `src/commands/config/show.ts` — emits resolved config with the
  token redacted (`MONDAY_API_TOKEN: <set>` / `<unset>`,
  **never** the value).
- `src/commands/config/path.ts` — emits the `.env` file path(s)
  considered.
- `src/commands/cache/list.ts`, `…/clear.ts`, `…/stats.ts`.
  `clear --board <bid>` deletes one board entry; without flag,
  clears everything.
- `src/commands/schema/index.ts` — emits the full CLI schema as
  JSON Schema 2020-12. `monday schema <command>` narrows.
  Schema source is a single `commandRegistry` keyed by command
  name; each command exports its zod input schema and a static
  output-shape descriptor.
- `cli/index.ts` registers all of the above.

**Tests (pyramid)**
- Unit: schema emitter produces valid JSON Schema 2020-12 for a
  representative command (use a JSON Schema validator); cache
  read rejects mode-`0640`; cache TTL boundary; config redaction
  retains structure.
- Integration: invoke each command via the in-process command
  invoker, assert envelope shape and stable error codes for
  failure modes (cache directory unwritable → `cache_error`,
  unknown command for `monday schema xyz` → `usage_error`).
- E2E: spawn `dist/cli/index.js`:
  - `monday schema --json` produces valid JSON.
  - `monday config show --json` redacts the token.
  - `monday cache list --json` works on an empty cache.

**Exit**
- All M1 commands listed in `--help`.
- `monday schema` describes itself (introspection round-trip).
- Coverage ≥ 90% on `src/api/cache.ts` and
  `src/commands/{config,cache,schema}/**`.

### M2 — API client + `account` commands

First network calls. Validates the entire I/O stack — error
mapping, retries, request_id propagation, complexity surfacing —
against the simplest possible commands.

**Deliverables**
- `src/api/client.ts` — `MondayClient` over the injected
  `Transport` interface (the M0 seam). Constructor takes
  `{transport, signal, retries, verbose}`. Sets `Authorization`
  (no `Bearer ` prefix) and `API-Version` (default `2026-01`,
  override via `--api-version` / `MONDAY_API_VERSION`) — header
  injection lives in `FetchTransport`, locked down per the
  Codex M0 §1 follow-up. Exposes typed wrappers for the
  operations v0.1 needs (whoami, account, versions,
  complexityProbe) **plus** a `raw<T>(query, vars)` escape.
  Internal callers see `Record<string, unknown>` for vars; the
  SDK's `QueryVariables = Record<string, any>` boundary leak
  is contained at this module's edge.

  **Deviation from the first plan draft.** The original wording
  said "wraps `@mondaydotcomorg/api`'s `ApiClient`". The SDK's
  `ApiClient.request` builds its own `AbortController` per call,
  which means the runner's SIGINT-aware `ctx.signal` can't
  cancel an in-flight request — and tests need a transport
  injection seam without monkey-patching SDK internals (Codex
  M0 §2). Both problems are solved by the existing `Transport`
  interface; the SDK still earns its keep through *types*
  (`Account`, `User`, `Complexity`, `Version`) and the
  `AvailableVersions.CURRENT_VERSION` constant. No SDK *method*
  runs at runtime.
- `src/api/errors.ts` — maps GraphQL / HTTP / network failures
  to the 24 M2-reachable error codes:
  - GraphQL `errors[0].extensions.code` →
    `unauthorized` / `forbidden` / `validation_failed` /
    `complexity_exceeded` / `rate_limited` /
    `daily_limit_exceeded` / `concurrency_exceeded` /
    `ip_rate_limited` / `resource_locked` / `not_found` /
    `stale_cursor`.
  - HTTP 423 → `resource_locked` (with `Retry-After`).
  - HTTP 429 / 401 / 403 / 404 / 5xx → status table.
  - **Body's top-level `error_code` outranks bare HTTP status**
    so a 429 carrying `IP_RATE_LIMIT_EXCEEDED` maps to
    `ip_rate_limited`, not `rate_limited` (Codex M2 review §3).
  - `AbortError` → wrapped abort (runner decides 130 vs 2).
  - `TimeoutError` → `timeout`.
  - Network failures → `network_error` with a generic,
    URL-free message (Codex M2 second-pass review §1).
  - Anything unmapped → `internal_error` with the original
    error as `cause` and a "report this" hint.
- `src/api/retry.ts` — exponential backoff + jitter for codes
  marked `retryable: true` (max retries from `--retry`,
  default 3). Honours `retry_after_seconds` when present.
  Since `MondayClient` calls `Transport` directly (not the SDK),
  the SDK's own retry layer (none in `graphql-request@6.1.0`,
  but the door's open for a future bump) **cannot** double-count.
  The regression test asserts exactly N=retries+1 transport
  attempts so a future SDK update that re-introduces a retry
  layer fails loudly.
- `src/api/complexity.ts` — opt-in `complexity { used,
  remaining, reset_in_seconds }` selection on every query when
  `--verbose`. Surfaces in `meta.complexity`.
- `src/commands/account/whoami.ts` — `me { id, name, email,
  account: { id, name } }`.
- `src/commands/account/info.ts` — `account { id, name, slug,
  plan, country_code, ... }`.
- `src/commands/account/version.ts` — emits
  `{ pinned, available }` (latest from API + the SDK's
  `CURRENT_VERSION`).
- `src/commands/account/complexity.ts` — uses the
  `complexity` query without a payload (cheapest possible).
- `src/utils/output/table.ts` gains an account-specific
  rendering branch (or stays generic).

**Tests (pyramid)**
- Unit: error mapper maps every Monday error code to the right
  CLI code; retry layer respects `retry_after_seconds`; client
  injects `API-Version` + `Authorization` headers; redaction
  test with token in `error.cause`.
- Integration: each `account` command via the injected
  `Transport` with a fixture cassette; asserts request body,
  headers (`Authorization`, `API-Version` injected by
  `FetchTransport`), response → envelope.
- E2E: ships a tiny in-process HTTP fixture server
  (`tests/e2e/fixture-server.ts`) that loads cassettes from
  `tests/fixtures/`. Spawn binary with
  `MONDAY_API_TOKEN=fixture` + `MONDAY_API_URL` pointing at
  the local server; assert exit codes 0 / 2 across
  success / failure paths.

**Exit**
- All four account commands ship with envelope + tests.
- HTTP fixture server runs in CI as part of the E2E suite.
- Every entry in §6.5's error code table has at least one test
  that produces it (most via injected mock errors). See §5.6
  for the exhaustive code-by-code map.
- `--verbose` adds `meta.complexity` to a real query.
- Token never appears in any error message, log, or stdout/stderr
  output. Verified by a hardened redaction regression suite:
  asserts the literal token (`"tok-leakcheck-xxxx"`) is absent
  from every emitted byte across the integration suite **and**
  exercises adversarial shapes — token in `Error.cause`,
  non-enumerable `Error.message`/`stack`, mixed-case header
  keys, arrays, circular references, debug/`--verbose` output,
  redacted log payloads.
- `npm pack` smoke-test runs in CI starting from M2 (lifted
  from M7 per Codex review §7) — the published tarball must
  contain a working bin entry.
- Coverage ≥ 90% on `src/api/**` and `src/commands/account/**`.

### M2.5 — Pre-M3 refactor pass

Structural-debt clean-up. M2 shipped enough infrastructure that
the trajectory is clear: M3 lands 14 commands, M4 lands 5 more,
M5b lands 4 more — **23 network commands** in three milestones, all
on the same skeleton. Three patterns from the M2 build will
calcify if they don't get tightened first; two more are flagged
as candidates we're choosing *not* to do yet, with explicit
revisit triggers.

**Why this milestone, not "do it during M3".** Refactor work
mixed with feature commits muddies the diff and doubles the
review surface. Land it as its own commit cluster, gate M3 on
it, and M3 reviewers see only feature changes against a
known-good baseline.

**Deliverables (do these — do not start M3 until they're in)**

- **R1. Move `src/commands/account/client-helper.ts` → `src/api/network.ts`**
  (or `src/api/client-builder.ts` — bikeshed at implementation time).
  Today the helper that resolves `loadConfig + globalFlags +
  transport + setMetaHint + MondayClient` lives under one noun
  even though every M3+ network command needs it. Moving it to
  `src/api/` puts it next to `Transport` + `MondayClient` and
  removes the cross-noun import M3 commands would otherwise need.
  Cheap now (~5 file edits, 4 imports updated, no behaviour
  change); painful once M3's 14 commands depend on it from across
  the tree. Done when:
  - new file lives at `src/api/<name>.ts`;
  - `src/commands/account/{whoami,info,version,complexity}.ts` import
    from the new location;
  - no `commands/<noun>/` file imports from `commands/<other-noun>/`;
  - typecheck + lint + test all green.

- **R2. Split envelope emission out of `src/cli/run.ts`**
  into `src/cli/envelope-out.ts` (or similar). The runner today
  owns: argv → commander, request-id, signal combining, command
  registration with three hooks (registry / extras / raw),
  MetaHint slot, error-envelope building, secret collection,
  signal-reason mapping. ~330 lines and growing — M3 will add
  cache-source tracking (`source: 'cache' | 'mixed'`); M4 will
  add warnings aggregation; M5b will add side-effects. Without
  the split, run.ts becomes a god-object somewhere around M5.
  The natural seam: anything that builds or writes an envelope
  (success or error) moves out; the runner keeps the
  argv/signals/command-registration core.

  **Redesign opportunity bundled into R2 (do not skip).** The
  current `setMetaHint(hint)` callback is a mutable side-effect
  channel into otherwise-pure error handling — added late in
  M2 to fix Codex review §2 (commit `5e211bc`). It works, but
  the M2-era patch shape is a wart: actions write into a
  hidden slot the runner reads on the error path, which means
  reasoning about "what `meta` will the error envelope carry"
  requires tracing every `setMetaHint` call site. Splitting
  envelope emission is the *one-time* chance to clean this up
  before M3 adds cache-source tracking + M4 adds warnings to
  the same channel. The replacement should be a real
  interface, not a setter — e.g. the new envelope-out module
  exposes a `MetaBuilder` (or returns a `(action) => Meta`
  closure) that commands narrow as they go, and the error
  path closes over the *same* builder rather than reading a
  mutable slot. Action sites stay as terse as the current
  `ctx.setMetaHint({apiVersion, source: 'live'})` — the
  difference is the data flow stops being a side-effect
  channel.

  Done when:
  - `cli/run.ts` is ≤ 220 lines;
  - new file owns `buildBaseMeta`, `writeErrorEnvelope`, and
    `collectSecrets`;
  - the new module has its own unit test file (`tests/unit/cli/
    envelope-out.test.ts`) — pure functions, easier to test than
    via the runner;
  - the `setMetaHint` mutable-callback pattern is replaced by
    a `MetaBuilder`-style interface (or equivalent — the shape
    is bikeshedable, the requirement is "no mutable hint slot
    on `ctx`"). If the implementer concludes after a real
    attempt that the current shape is genuinely the best
    available, document the alternatives considered + why in
    a top-of-file comment in the new module + a row in §13
    M2.5 post-mortem;
  - existing `tests/unit/cli/run.test.ts` still green; no
    behaviour change at the public boundary;
  - `tests/integration/commands/account.test.ts >
    "--api-version is also reflected in the error envelope
    meta"` still pins the contract (the test that originally
    drove the M2 fix — must survive any redesign).

- **R3. Compact `emitSuccess` option surface**
  by introducing an `EmitFromNetworkResult` (or similar) shape
  that bundles `{source, apiVersion, complexity, cacheAgeSeconds}`.
  Today `emitSuccess` accepts 10 named options; four of them
  always come from a single `resolveClient` + `client.<method>`
  pair. Letting the helper return a shape that `emitSuccess`
  can splat directly compresses each command's emit call from
  ~10 lines to ~5 and removes the per-call ceremony of
  threading `apiVersion` + `complexity` + `source` separately.
  Tradeoff: another small intermediate type. Cheap because the
  type is internal — no contract change, no schema drift.
  Done when:
  - the four account commands' `emitSuccess` calls are ≤ 6
    arguments each;
  - the `EmitFromNetworkResult` interface lives wherever
    `resolveClient`'s return type now lives;
  - integration tests still green; no envelope shape change.

**Tracked but deferred (do NOT ship in M2.5; revisit per trigger)**

At each revisit, if the decision is **not** to ship, append a one-
line update to the relevant row below recording what the data showed
and any revised trigger. Future agents shouldn't re-litigate the
same deferral with stale context — and the call may need to be made
multiple times across milestones before the data is decisive.

- **R4. `defineNetworkCommand({...})` factory.**
  Compresses the per-command `parse argv → resolveClient → call
  typed method → null-check → emit` skeleton from ~30 lines to
  ~10. M3 has 14 commands, M4 has 5, M5b has 4 — by the time M5b
  ships there will be ~23 instances of the same skeleton. But:
  - the explicit form catches per-command quirks (whoami's
    `me === null → unauthorized`, info's
    `account === null → not_found`, complexity's
    `c === null → internal_error`) at the right granularity;
  - factories hide control flow, which the M0-era design intent
    explicitly rejected ("per-command differences are best
    captured as code in `attach` rather than ferried through
    shared options that grow unbounded");
  - the right time to extract is *after* M5b, when 23 real
    examples + the mutation-shape (which M2 doesn't have) make
    the right shape obvious.
  - **Revisit trigger:** post-M5b. If by then the boilerplate
    is rote and ≥ 80% identical across commands, factor it
    out. If per-command quirks dominate, leave it.
  - **Update (post-M3):** §15 R7 ships a narrower, get-by-id-only
    version of this. If R7 captures the bulk of the duplication
    when it lands, the broader R4 factory may not be worth
    pursuing — re-evaluate at the post-M5b revisit with R7 already
    in place.

- **R5. Cassette match-logic dedup** between
  `tests/fixtures/load.ts` (in-process FixtureTransport) and
  `tests/e2e/fixture-server.ts` (out-of-process HTTP server).
  ~50 lines overlap (`queryMatches`, `variablesMatch`,
  `headersMatch`). Two consumers isn't a strong case for an
  abstraction, and the request shapes differ slightly
  (Transport-shape vs HTTP-shape) so a shared `match(spec, req)`
  needs an adapter layer. Premature.
  - **Revisit trigger:** if a third cassette consumer appears
    (most likely candidate: a future `monday raw` smoke test in
    M6, or a live-mode replay in v0.2). Until then, the dup is
    cheaper than the abstraction.

### M4 — `item` reads + filtering + pagination

The heaviest read wave. Filter DSL leans on M3's column
read-resolver; pagination contract gets exercised here.

**Deliverables**
- `src/api/filters.ts` — parses `--where <token><op><val>`.
  Operator allowlist: `=`, `!=`, `~=`, `<`, `<=`, `>`, `>=`,
  `is_empty`, `is_not_empty`. Token resolves via
  `src/api/columns.ts` (M3) — supports column ID, title,
  `id:`/`title:` prefixes, and the `me` sugar for people
  columns. Emits a `query_params.rules` object suitable for
  `items_page`. `--filter-json` is passed through verbatim.
  Multiple `--where` flags are AND'd. Edge cases the unit
  tests must cover: column titles containing operators
  (`Plan A=B`), Unicode normalisation collisions,
  `due:is_empty` (no value), `me` in people-column `--where`,
  column titles with leading/trailing whitespace.

  age tracked via the injected clock so tests can replay
  expiry without wall-clock waits.
- `src/api/sort.ts` — per-page sort by ID asc (deterministic
  per §3.1 #8). Documents that cross-page order is Monday's
  unless `--filter-json` includes an `order_by`.
- `src/commands/item/list.ts` — `--board <bid>` required,
  optional `--group`, `--where` (repeatable), `--filter-json`,
  `--state active|archived|all`, `--all`, `--limit`.
- `src/commands/item/get.ts` — `items(ids: [<iid>])`; full
  column values per §6.2.
- `src/commands/item/find.ts` — `--board <bid>` required;
  uses `findOne` resolver against item names.
- `src/commands/item/search.ts` —
  `items_page_by_column_values` with the same `--where`
  parser.
- `src/commands/item/subitems.ts` — list children.

**Spec gaps surfaced post-implementation (backfill candidates).**
Same shape as M3's `--limit-pages` / `pagination_cap_reached`
surface gap. None blocks M4 shipping; each is a contract-clarifying
PR for cli-design.md when next opened.

- *`--state` was deferred from `item list`.* The spec's M4
  deliverable lists `--state active|archived|all`, but Monday's
  `2026-01` `items_page` doesn't accept a top-level state arg, and
  client-side filtering across multiple pages would silently shrink
  result counts mid-walk. The flag was dropped from the runtime;
  archived-item listings ship when a future milestone wires
  `archived_items_page`. cli-design.md still mentions the flag in
  §4.3 prose — backfill on next design revision.
- *`title:`/`id:` prefix is brittle for binary-operator titles.*
  Per §5.3 step 2.b, `parseWhereSyntax` splits on the first
  operator, so `--where 'title:Plan A=B=approved'` resolves the
  token to `title:Plan A` (via the parser's first-`=` split)
  rather than honouring the explicit-`title:`-prefix intent. The
  documented escape hatch for operator-in-title columns is
  `--filter-json`. Two follow-up options: (a) add an unambiguous
  quote/escape grammar to the `--where` parser; (b) tighten the
  `title:`/`id:` prefix semantics to "scan for operator after the
  prefix's value boundary". Neither lands in M4 — the
  `--filter-json` path covers the case today; agents pasting raw
  Monday filter rules already work without extra surface.
- *`pagination_cap_reached` warning shape on `item find`.* M3
  introduced `pagination_cap_reached` for page-based walks
  (`workspace list --all --limit-pages`). M4 reuses the warning
  code on `item find` when the cap-bounded scan was truncated and
  uniqueness can't be verified; details schema is parallel
  (`pages_scanned`, `cap_pages`, `items_scanned`, `hint`). Backfill
  cli-design.md §6.5 / §5.7 to document the find-cap variant on
  next design revision.

  use `change_simple_column_value` and rich types use
  `change_column_value`; multi bundles into
  `change_multiple_column_values`. What it doesn't say is that
  the per-column blob inside the multi mutation requires
  `{text:<value>}` for `long_text`, even though
  `change_simple_column_value(value: String!)` accepts the bare
  string for the same column. `text` and `numbers` stay as bare
  strings inside multi; only `long_text` gets re-wrapped.
  selectMutation pins the wire shape via fixture and the
  divergence is documented in the module header. **For
  cli-design §5.3 step 5 backfill** — either explicitly call
  out the long_text re-wrap, or move the per-column-blob
  shapes into a sub-table. Codex pass-1 confirmed Monday's
  changelog/docs don't directly contradict the
  `{text:...}` shape; live verification will pin it once
  M5b spawns its first integration cassette.
- **Safe-integer bound on status indexes / dropdown IDs.** The
  translators throw `usage_error` when numeric input exceeds
  `Number.MAX_SAFE_INTEGER` (2^53 - 1). cli-design §5.3 step 3
  describes the index/ID payload shapes but doesn't bound the
  input range. Implementation guard around JS number safety,
  not a contract gap — Codex pass-2 explicitly *declined* to
  log this as a spec gap, but it's worth knowing the helper
  is opinionated about input range. v0.2 may revisit if any
  Monday board surfaces an integer-shaped label or ID > 2^53
  (vanishingly unlikely; auto-incrementing IDs would take
  centuries at Monday's API rate to climb that high).
- **Dropdown empty-input semantics.** `--set tags=""` throws
  `usage_error` (the helper doesn't have an item ID, so the
  hint message uses placeholder `<iid>` syntax pointing at
  `monday item clear <iid> <col>`). cli-design §5.3 doesn't
  spell out empty-string handling; the convention adopted
  here keeps `--set` and `monday item clear` non-overlapping.
- **Date relative-offset bound (commits `b9efef8` /
  `53e0eb0`).** Relative tokens (`+Nd`, `+Nw`, `+Nh`) are
  bounded to ±100 years magnitude — `MAX_RELATIVE_DAYS = 36500`
  and `MAX_RELATIVE_HOURS = 876000`. cli-design §5.3 step 3
  enumerates the relative-token forms but doesn't bound the
  amount range. Implementation guard against JS Date overflow
  (±100M days from epoch) — pre-bound, `+99999999d` produced
  `{date:"0NaN-NaN-NaN"}` and unsafe hour values could
  RangeError instead of UsageError. Worth folding into the
  cli-design §5.3 grammar prose so agents who hit the bound
  understand the limit is documented, not arbitrary.
- **Date grammar accepts case-insensitive relative tokens +
  trims surrounding whitespace + accepts `HH:MM:SS` seconds
  (commit `30402b8`).** cli-design §5.3 step 3 lists `today`,
  `+3d`, `-1w`, `+2h` and the date+time form
  `2026-04-29T14:30 → time: "14:30:00"`. The implementation
  also accepts `TODAY`, `Today`, `tOdAy` (case-folded for
  matching, `resolvedFrom.input` echoes verbatim);
  whitespace-trimmed inputs (`" today "` resolves the same
  as `today`); and `HH:MM:SS` with explicit seconds
  preserved. None of these contradict the design — they're
  agent-friendly extras — but worth a one-sentence backfill
  so the grammar's surface is fully documented.
- **Date ISO year policy (commit `b9efef8`, Codex pass-2
  spec-gap note).** The regex accepts any 4-digit year
  `0000`-`9999`. `setUTCFullYear` validation accepts pre-1900
  years (post-pass-1 fix); 5-digit years and BCE forms are
  rejected at the regex layer. cli-design §5.3 step 3
  doesn't enumerate the year range. Translator's contract
  is "let Monday be the validator" — Monday rejects ancient
  dates as `validation_failed`. Worth a one-sentence
  backfill so agents who pass `0001-01-01` know the
  rejection comes from Monday, not the CLI.
- **`me` token casing parity across all three surfaces
  (commit `4c97584` Codex pass-1, `6429cd7` Codex pass-2).**
  cli-design §5.3 step 3 line 704-707 says "`me` resolves to
  the connected user's ID" without specifying case-
  sensitivity. Pre-fix, three surfaces handled `me` slightly
  differently: `--where Owner=me` (filter, `filters.ts`) and
  `item search --where Owner=me` (`commands/item/search.ts`)
  were case-sensitive, while `--set Owner=me` (`api/people.ts`)
  was case-insensitive. Pass 1 unified filter; pass 2 unified
  search. All three now lower-case the trimmed token before
  the equality check — `me` / `ME` / `Me` / `mE` resolve
  identically, mirroring the design's "me sugar" intent. The
  divergence between read and write surfaces was the spec
  gap; the unified rule is the new contract. Worth a
  one-sentence backfill on cli-design §5.3 line 704-707
  ("case-insensitive after trim").
- **People grammar — numeric tokens rejected, no team
  support, comma-split semantics (commit `16c6bf7`).**
  cli-design §5.3 step 3 line 728-734 enumerates `me` and
  comma-split emails for the people grammar but doesn't
  bound:
    1. Numeric tokens (`--set Owner=12345`). Implementation
       rejects with `usage_error` carrying
       `deferred_to: "v0.2"` (M5b post-mortem Path B replaced
       the original literal `--set-raw` paste-ready hint, since
       v0.1 has no `--set-raw` flag). Agents with a raw user
       ID look up the email and use the email form, or wait
       for v0.2's `--set-raw`. Edge cases pinned via test:
       signed numerics (`+12345`) fall through to
       `resolveEmail` (regex is `^\d+$`, sign-free), so they
       surface as `user_not_found` after the directory lookup
       fails. Decimal numerics (`1.5`) and scientific (`1e3`)
       fall through similarly.
    2. `kind: 'team'` support. Monday's API accepts
       `personsAndTeams[].kind` as `'person' | 'team'` for
       team-assignment columns, but cli-design only lists
       `'person'`. v0.1 implementation hard-codes the
       literal `'person'`; teams are a v0.2 candidate that
       lands alongside `--set-raw` in the writer-expansion
       milestone. Until then a team column surfaces
       `unsupported_column_type` with `deferred_to: "v0.2"`.
    3. Empty-segment dropping behaviour (`alice@example.com,
       ,bob@example.com` → two-entry payload; `--set
       Owner=""` → `usage_error`). Same shape as the
       dropdown empty-input branch — non-overlapping with
       `monday item clear`.
- **Resolver-side user-ID schema tightening (commit
  `6429cd7` Codex pass-2 F4).** `userByEmail`'s schema was
  `id: z.string().min(1)`, which silently let malformed IDs
  (`"0x2a"` / `"1e3"` / `"42 "`) into the directory cache
  where `Number(id)` would corrupt every consumer's wire
  payload. Tightened to `^(0|[1-9]\d*)$` (decimal
  non-negative integer string). Same regex shape lives in
  `api/people.ts` as `DECIMAL_NON_NEGATIVE` for defence in
  depth — both layers reject malformed IDs with
  `internal_error`. cli-design §6.5 doesn't enumerate the
  data-integrity-from-resolver case explicitly, so the new
  `internal_error` surface here is consistent with the
  catch-all but worth flagging that a resolver-data error
  is now a recognised internal_error trigger.
- **R-JsonValue refactor (commit `086b34a`).** Codex review
  pass 1 on the status/dropdown commit (`f932a90`) flagged
  that `Readonly<Record<string, unknown>>` admits non-JSON
  values (`undefined`, symbols, functions, class instances).
  Deferred at the time; landed during the people session
  because people was the first translator with a nested array
  of typed objects. Tightening replaced 5 references in
  `column-values.ts` with `JsonObject` (defined in
  `src/types/json.ts`). The closed-type-literal limitation
  (closed object types like `PeoplePayload` don't implicitly
  satisfy open index signatures) means the people dispatch
  needs an `as unknown as JsonObject` cast at the boundary;
  documented at the cast site. Codex pass-2 noted that
  `JsonValue` doesn't catch `NaN` / `±Infinity` / cycles /
  symbol keys / BigInt — TS structural narrowing limits;
  documented in the module header. Not a contract gap, just
  a useful awareness boundary.
- **People-resolution echo shape (commit `9dddf48`).** The dry-run
  engine renders `details.resolved_from` for people inputs as
  `{tokens: [{input, resolved_id}, ...]}` per cli-design §6.4
  (which only shows the date case). The token-by-token shape
  pairs each comma-split input with its resolved Monday user ID
  (string form to round-trip safely through JSON.stringify for
  high-range IDs). `resolved_id` is a string even though the wire
  payload uses numbers — the echo doesn't need to round-trip a
  number, and the directory schema's shape is the source of truth.
  Worth a one-paragraph backfill on cli-design §6.4 to enumerate
  the per-kind details shapes (date vs people) so future
  translators know which slot to populate.
- **`details.resolver_warnings` on `column_archived` errors
  (commit `26ce927` Codex pass 2 F1).** When a stale-cache refresh
  resolved to an archived column, the `stale_cache_refreshed`
  warning was lost (success-path warnings array → typed error
  envelope which has no `warnings` slot). The fix folds resolver
  warnings under `error.details.resolver_warnings` so the agent
  sees both signals — cache was stale AND target was archived.
  cli-design §6.5 doesn't enumerate per-error `details` slots;
  worth a one-line backfill for the `column_archived` error so
  agents who read `details.resolver_warnings` know the slot is
  documented.
- **Echo-slot exclusivity invariant (commit `26ce927` Codex pass 2
  F3).** `TranslatedColumnValue` carries two parallel echo slots
  (`resolvedFrom` for date / `peopleResolution` for people). v0.1
  translators populate exactly one or neither; `buildDiffCell`
  fires a typed `internal_error` if both slots are set
  simultaneously (a translator wiring bug). Logged here because
  the invariant + the `internal_error` surface are both
  implementation-only — cli-design §6.4 only specifies the
  user-visible `details.resolved_from` shape, not the translator's
  internal echo plumbing. Worth keeping as the "Option B
  documentation footprint" in case Option A (discriminated union)
  becomes the v0.2 widening when a third echoing translator lands.

## 4. Cross-cutting "definition of done" per command

Every command, regardless of milestone, must satisfy:

1. **Source.** One file in `src/commands/<group>/<verb>.ts` for
   noun-verb commands; `src/commands/<noun>/index.ts` for
   noun-only commands (`schema`, `raw`). Imports `api/`, never
   the SDK directly. Top-of-file comment names the GraphQL
   operation(s) it calls (per `cli.md`).
2. **Schemas.** Exports an `inputSchema` (zod) for argv parsing
   **and** an `outputSchema` (zod) describing the `data`
   payload. The same `outputSchema` powers `monday schema`
   (via `z.toJSONSchema`) and the envelope contract test —
   no separate static descriptor.
3. **Args parsed via zod before any API call.** Commander
   delivers `string`/`string[]`/`boolean`; the zod schema
   coerces and refines at the command's edge. Internal code
   sees the validated type.
4. **Help text.** `description()` + at least one usage example
   in `addHelpText('after', ...)` (per `cli.md`).
5. **Stdin contract** documented if the command accepts stdin
   (e.g. `--body-file -`, ID-from-stdin via `xargs` pattern).
6. **Idempotency.** Documented: yes/no in the command
   description. **Every mutation supports `--dry-run`** —
   including non-idempotent ones like `update create`, where
   the dry-run reports "would post comment X to item Y".
7. **Tests.**
   - Unit covering every branch of the command's pure logic.
   - Integration covering happy path + every error code the
     command can produce + format variations + envelope
     contract test (per §5.2).
   - E2E if the command is non-trivial (any mutation, any
     paginated read, any command with stdin or file inputs).
8. **Output.** Every applicable format produces sensible
   output: `json` and `table` always; `text` for
   single-resource commands; `ndjson` for collections that
   benefit from streaming (`item list --all`). Non-applicable
   formats fail with `usage_error` (e.g. `--ndjson` against a
   single-resource command). Table truncation is honoured;
   JSON / NDJSON never truncate. Snapshot-tested.
9. **Stdout/stderr discipline.** Result on stdout (or
   nothing-on-stdout for an error). Errors, warnings (in TTY
   mode), debug logs all go to stderr. The envelope contract
   test asserts no stdout pollution on failure.
10. **Errors.** Every thrown error inherits a typed class with
    a `code` from §6.5. No plain `throw new Error(...)` in
    `commands/`.
11. **No token leakage.** Verified by the redaction regression
    suite (M2).

## 5. Testing strategy — the pyramid, made explicit

The project's testing rule is "every branch covered" (`testing.md`),
not just "hit the 90% threshold". The pyramid below is how we
distribute that effort across layers.

```
              ┌────────────┐
              │    E2E     │   ~5–10% of tests
              │ (binary)   │   broad smoke + 1 agent flow
              └────────────┘
           ┌──────────────────┐
           │   Integration    │   ~20–30%
           │ (mock transport) │   one per command, all error codes
           └──────────────────┘
        ┌────────────────────────┐
        │         Unit           │   ~60–75%
        │   (pure functions)     │   every branch, every error
        └────────────────────────┘
```

### 5.1 Unit (`tests/unit/**`)

- **Scope.** Pure logic only. `src/utils/**`, `src/types/**`,
  `src/api/{errors,filters,column-values,dry-run,...}.ts`,
  per-command argument parsers and formatters. No network. No
  fs writes outside `os.tmpdir`.
- **Coverage rule.** Every branch named in `testing.md` —
  happy / edge / error / format variations. The 90% v8
  threshold is a floor; the goal is "every named branch hit
  by an `it` block".
- **Mocking.** Vitest `vi.fn()` for collaborators *only* when
  the test's purpose is to verify the call. Prefer real
  inputs and assert real outputs.
- **Naming.** `<file-under-test>.test.ts`, colocated under
  `tests/unit/<mirror-of-src-path>/`. `describe` blocks name
  the function/class; `it` blocks complete "it ___".

### 5.2 Integration (`tests/integration/**`)

- **Scope.** A whole command, end-to-end through the
  `commands/` → `api/` → injected `Transport` stack. Every
  command in v0.1 has at least one integration test.
- **Mocking strategy.** Inject a `FixtureTransport` via
  `run({ transport })` (M0). The transport implements the
  same interface as `FetchTransport` so header injection,
  retry, abort handling, and error mapping all run unchanged
  — the only difference is where the bytes come from. **Do
  not** spy on internal SDK methods or monkey-patch
  `commands/*` helpers; that bypasses the layers we want to
  exercise.
- **Cassette format** (per Codex review §2 — single
  request/response pairs aren't enough). Each fixture is an
  ordered list of interactions:

  ```ts
  interface Cassette {
    interactions: Interaction[];
  }

  interface Interaction {
    operation_name?: string;        // fast match key
    match_query?: RegExp | string;  // optional substring/regex
    match_variables?: Partial<Vars>;
    expect_headers?: Record<string, string | RegExp>;
    response?: { data?: unknown; errors?: GraphQLError[] };
    http_status?: number;           // for non-200 (e.g. 423)
    delay_ms?: number;              // for timeout tests
    repeat?: number;                // for paginated walks
  }
  ```

  Stored at `tests/fixtures/<group>/<verb>/<case>.json`.
  Loader in `tests/fixtures/load.ts`. Multi-page walks and
  retry sequences are first-class.
- **Required cases per command.** Happy path + every error
  code reachable from the command + each `--output`
  variation that affects rendering + each significant flag
  combination (e.g. `--all`, `--limit`, `--filter-json`,
  `--dry-run`, `--no-cache`).
- **Envelope contract tests.** A shared test helper
  validates *every* emitted JSON envelope across the suite:
  `meta` skeleton present, `meta.schema_version === "1"`,
  errors only on stderr, no stdout pollution on failure,
  emitted output validates against the command's
  `outputSchema`. Every command's integration test runs
  through this helper before its case-specific assertions.
- **Forbidden.** Real network. Spawning the compiled binary.

### 5.3 E2E (`tests/e2e/**`)

- **Scope.** Spawn `node dist/cli/index.js …` via
  `child_process.spawn`. Network mocked by an in-process
  HTTP fixture server (`tests/e2e/fixture-server.ts`,
  shipped in M2) listening on a random port; spawn with
  `MONDAY_API_URL=http://127.0.0.1:<port>`. The server
  loads the same cassettes as the integration suite, so
  fixtures are not duplicated. Assert on `stdout`,
  `stderr`, exit code.
- **Required cases.**
  - Per noun: at least one happy-path command.
  - Per exit code: 0 (success), 1 (usage — bad flag),
    2 (API — fixture rejection), 3 (config — missing token),
    130 (SIGINT, via spawn-and-kill).
  - Per output format: `--json`, `--table` (force TTY),
    `--ndjson` for `monday item list --all`.
  - The agent flow from `examples.md` §1 (M6).
- **`RUN_LIVE_TESTS=1`** opt-in tier hits the real Monday
  API against a dedicated test workspace. Not run in CI by
  default. Each live test must clean up everything it creates
  (`try { ... } finally { cleanup() }`).

### 5.4 Coverage targets

- `vitest run --coverage` includes unit + integration. E2E
  spawns a separate process so v8 doesn't see it; we accept
  that.
- Threshold floor (configured): **90% lines, branches,
  functions, statements**. Raise as the codebase grows; do
  not lower.
- The threshold is necessary but not sufficient. Per-command
  PRs include a manual branch-coverage checklist (happy /
  edge / error / format) that reviewers can audit.
- `tests/fixtures/` is excluded from coverage.

### 5.5 Anti-patterns to avoid

- **Internal-call snapshots.** Don't snapshot the mocked
  transport call argument unless the test's whole purpose is
  to verify the GraphQL request body — and then do it
  explicitly with `toMatchObject`, not a freeform snapshot.
- **Shared state.** Each `it` is independent. Setup in
  `beforeEach` (not `beforeAll` unless mutation-free).
- **Skipping the unhappy path.** Every command's integration
  test covers at least one error path. PRs that skip
  unhappy-path tests get sent back.

## 6. Risk register

Risks ranked by impact × likelihood (top = most pressing).
Codex review §3 prompted the re-rank — token leakage to the
top, SDK retry downgraded, three new risks added.

| Risk | Impact | Mitigation |
|------|--------|------------|
| **Token leakage** — token in `Error.cause`, non-enumerable `Error.message`/`stack`, header casing variants, debug logs, retried-request URL. | **High** | All output paths funnel through `redact()`. ## 8. Decisions (open questions closed)

These were open in the first draft of this plan; the Codex
review §7 surfaced enough evidence to close all five. The
decisions are recorded here so M0 doesn't relitigate them.

1. **Table renderer: `cli-table3`.** Small, well-tested
   dependency; less renderer code to own than the
   alternative. Adding it in M0.
2. **JSON Schema emission: zod v4 native `z.toJSONSchema()`.**
   The installed `zod` version exposes it. Tests verify
   brands, refinements, and coercions round-trip via the
   emitter (a known sharp edge with zod-to-JSON Schema
   libraries).
3. **Cassette format: hand-written, ordered multi-interaction.**
   See §5.2 for the schema. Single request/response pairs are
   too thin for paginated walks and retry sequences; the
   ordered shape is first-class from the start.
4. **Bin entry: keep `dist/cli/index.js` as `bin`.** TypeScript
   preserves the shebang. Move the `npm pack` round-trip
   smoke-test from M7 to M2 so the bundling surface gets
   exercised early — catches `package.json` `files`/`bin`
   regressions before they pile up.
5. **`--profile` in v0.1: accept absent or `default`.** Anything
   else returns `usage_error` with a structured `details.hint`
   pointing at v0.3. `MONDAY_PROFILE` is treated identically
   (env / flag must agree).

### Why M3 surfaced 5× as many findings as M2.5

M3 is the first milestone with a 14-command surface. Three structural
factors:

1. **Volume.** Every command is a chance for the same drift (parse
   boundary not wrapped, `--all` not capped, error-meta not
   committed). 14 commands × 3-4 drift shapes = ~50 audit points;
   even a 95% per-point hit rate leaves ~3 misses. Pass 1's first
   finding (no `--all` cap) hit *every* `--all`-walking command,
   not just one — fixing it required a shared helper.
2. **New surface area.** M2.5 was structural cleanup with no new
   wire shapes; M3 adds a flag (`--limit-pages`), a warning code
   (`pagination_cap_reached`), 3 new internal modules, and 14
   per-command shapes. New shape = new contract drift candidate.
3. **The cache boundary.** `loadBoardMetadata` hides a live-vs-cache
   distinction agents care about (complexity, source, cache age).
   The result object was originally optimised for the 80% case
   (read the metadata, ignore the meta); the 20% case (verbose
   complexity, mixed source, refresh warning) needed the result
   shape to grow. Two of pass-1's four findings + one of pass-2's
   three were variants of "this slot existed conceptually but not
   in the result type."

### Single load-bearing lesson

When a milestone introduces a meta-carrying boundary (cache,
resolver, paginator), the result type should carry **every meta
slot the §6.1 envelope can carry** from day one — even when the
first command using it doesn't read all of them. M3's
`BoardMetadataLoadResult` and `ResolveColumnWithRefreshResult` both
under-budgeted their meta surface; both grew during review.

The follow-up (which M4 should adopt prophylactically): when adding
a new shared module that produces network-derived data, the result
type's *first version* must include `source`, `cacheAgeSeconds`,
`complexity`, and `warnings: readonly Warning[]`. Every consumer
ignores the slots they don't need; future review-time additions
become non-events.

**Tests as a regression gate.** The `walkPages` cap test
(`tests/unit/api/walk-pages.test.ts`) explicitly asserts the cap
fires on a still-full page — it would fail against the pre-fix
unbounded loop. Same shape for the `mixed` cacheAgeSeconds test
(non-null) and the `complexity` propagation test. Each Codex
finding's fix landed with at least one test that would fail against
the pre-fix code; the repo should never accept a Codex-driven fix
that doesn't.

### Why M4 surfaced ~2× as many findings as M3

M4 is the first milestone with a *fundamentally new pagination
shape* (cursor-paginated `items_page` vs M3's page-based
`workspaces` / `boards` / `users` / `updates`) plus a *new DSL
parser* (the `--where` syntax + filter-JSON passthrough). Three
factors compound:

1. **New surface area, by volume.** 5 commands × (cache-aware
   metadata load × filter resolution × cursor walk × NDJSON-stream
   branch × envelope build) = ~25 audit points per command. Even at
   95% per-point hit rate, ~6 misses across the cluster — and pass
   1's 8 findings + pass 2's 5 are roughly that.

2. **Cursor pagination is genuinely subtler than page-based.**
   The §5.6 contract has three load-bearing rules (60-min lifetime,
   fail-fast on stale, never silently re-issue) plus per-call
   semantics (`limit:` matches what's emitted, not what's
   collected). Pass-2's blocker — over-fetching when `--limit <
   pageSize` — is a textbook example: the obvious implementation
   passes a fixed `pageSize` and trims client-side, which is
   correct for in-memory truncation but corrupts cursor state.
   This kind of subtle protocol-level invariant only surfaces in a
   second-pass review with the post-fix code in front of you.

3. **The cache-aware filter parser was a new boundary.** §5.3
   step 5 says column resolution must refresh metadata once on a
   cache-miss before surfacing `column_not_found`. The M3 single-
   token resolver (`resolveColumnWithRefresh`) implemented this for
   one token at a time; the M4 multi-clause filter parser had to
   re-implement the same semantic for an array of clauses with a
   *shared* refresh budget. Three of pass 1's eight findings + one
   of pass 2's five (`--api-version` round-trip on usage_error)
   were variants of "this contract held in M3 but didn't transfer
   to M4's new shape automatically".

### Single load-bearing lesson

When a milestone introduces a *protocol-level invariant* (cursor
state corruption, refresh-once budget, clock injection that has to
thread through every command), the per-command audit must look at
the *seam* between the shared module and the consumer, not just at
the shared module in isolation. M4's pagination.ts unit suite
covered the cursor walker correctly in isolation — but every M4
command that wrapped the walker had a chance to break the
invariant by passing the wrong fetchInitial shape, the wrong now
callback, or by ignoring the result's source/cacheAgeSeconds slots.
The pass-2 blocker (limit < pageSize over-fetch) is an example: the
walker did exactly what it was told; the *interface* between
walker and command was wrong.

The follow-up (which M5a should adopt prophylactically): when a
new shared module producing protocol-level invariants ships, write
the integration test that drives every wrapping command through
the boundary in the same commit, not the one after. M4's first
two integration test commits passed because the wrapping commands
*looked* correct; pass-1 review caught what only "drive the real
contract end-to-end" testing reveals.

**Tests as a regression gate.** Every pass-1 + pass-2 fix landed
with at least one test that fails against the pre-fix code.
Examples:

- `tests/unit/api/pagination.test.ts`: "shrinks the per-call limit
  to remaining budget" (pass-2 §1 — pre-fix walker requested
  pageSize=100; post-fix requests min(2, 100)=2).
- `tests/unit/api/pagination.test.ts`: "reports has_more=true when
  --limit truncates a terminal (cursor=null) page" (pass-1 §3 —
  pre-fix `finish()` saw cursor null and reported has_more=false).
- `tests/unit/api/filters.test.ts`: "fires onColumnNotFound once
  on cache-miss and emits stale_cache_refreshed warning" (pass-1
  §1 — pre-fix used pure resolveColumn).
- `tests/integration/commands/item.test.ts`: "reports source:
  mixed when board metadata is cached and items are live" (pass-1
  §2), "filter parser refreshes board metadata on cache-miss
  column lookup" (pass-1 §1 + §2), "drops per-cell column titles
  in collection output" (pass-1 §8), "emits pagination_cap_reached
  warning when find scan was capped" (pass-1 §6),
  "--api-version reaches the usage_error envelope on parseArgv
  failure" (pass-2 §3).

Same gate as M3: a Codex-driven fix without a regression test that
fails against the pre-fix code is not acceptable. Two of M4's
seven applied fixes (§4 + §5 from pass 2) are *intentionally*
documented rather than coded — the contract change vs the
documentation gap is the right shape for those.

### Why M5a was multi-session

The plan splits M5a into "produce the modules M5b consumes" — so
each session was a self-contained module + its tests + its review
pair. The benefit was clear: each module shipped with its full
review treatment before the next session started writing code on
top. The cost was overhead — six review prompt files, six "set up
the next session" handoff prompts, six "previous-session context
load" reads.

The pattern that worked: each session's first commit was a
deliberate **infrastructure pre-feature pass**. R8 opened the
status/dropdown session (writable allowlist + parseColumnSettings);
R-JsonValue opened the people session (rich-payload type
tightening); R15 + R16 + R17 opened the closing session (resolver
seam consolidation + parse-boundary wrap). The feature commit then
landed against an already-cleaned seam, which made review easier
to scope (the feature itself, not feature + cleanup) and made the
closing session's column-archived fix possible to reason about
without rewriting both the column resolver AND the dry-run engine.

### Single load-bearing lesson

**When a milestone has multiple translators that share a common
output type, treat the output type as a contract surface.** M5a's
`TranslatedColumnValue` started with `{columnId, columnType,
payload, rawInput}` — by the closing session it carried two
parallel echo slots (`resolvedFrom: DateResolution | null` and
`peopleResolution: PeopleResolution | null`). The Option A vs
Option B decision the closing session faced (extend `resolvedFrom`
to a discriminated union vs add a parallel slot) was a direct
consequence of the type's accreting per-translator details. The
right move WOULD have been to land a discriminator from the start
(`echo: { kind: 'none' } | { kind: 'date', ... } | { kind: 'people',
... }`) so the seventh translator was the one that widened the
union, not the closing session that paid for the lack of
discriminator. The current Option B parallel-slot shape works for
v0.1 — `buildDiffCell`'s explicit `internal_error` exclusivity
check guards the invariant — but a future translator adding an
echo kind faces a third parallel slot and the same revisit.

The follow-up (which M5b should adopt prophylactically): when a
new shared output type ships its first per-variant detail, land
the discriminator in that same commit, not as a future widen.
Three echo-needing types (date, people, hypothetical timeline)
becomes three parallel slots; one discriminated union with three
arms is structurally cleaner.

1. The runtime implementation (the error builder).
2. **All consumers of the runtime contract** — every error

3. **The binding contract doc** — cli-design.md's prose, every

   per-error-code surface in §6.5.
4. **Local contributor guidance** — module JSDocs, code comments

5. **Adjacent docs that quote the contract** — v0.1-plan.md

fix swept the rest. **Sweeping all five layers in one commit
makes the next round pass on first try.** The shape that works:
when a runtime change lands, write a follow-up commit that
greps for the OLD contract language and rewrites every match,
even when the test suite is already passing — the test suite
proves the runtime is correct, not that the docs match.

### Why M6's close arc ran 6 rounds

The **gate-correctness bugs were nested**: round 1 caught the
absence of any gate (P1 #1 — no mutation rejection at all). The
fix added the gate, which exposed round 2's pre-network
`meta.source` contract issue (the fix's new validation step
threw before `setSource('live')` made sense). The fix moved
`resolveClient`, which exposed round 4's `--dry-run` contract
gap (the new pre-`resolveClient` path didn't honour §9.2). The
fix added the dry-run branch, which exposed round 5's
gating-precision issue (the new branch's predicate was too
broad — `hasMutation` instead of `selectedOperationKind`).

Each round's findings were **caused by the previous round's
fixes**, not regressions of pre-fix bugs. That's a different
shape from M5b's three-round close, where round 1 was contract
drift and round 2 was doc drift downstream of the round-1
contract fix. M6 is the first close-arc where each round
introduced a *novel* bug class:

  1. **Round 1 → 2 (security/honesty):** new pre-network code
     path → exposed a latent meta.source contract bug.
  2. **Round 4 (contract):** new pre-network path → bypassed
     the `--dry-run` universal-mutation contract.
  3. **Round 5 (precision):** new dry-run branch → wrong
     predicate (document-wide vs selected-operation).

Lesson for v0.2: **adding a pre-`resolveClient` validation
step expands the failure surface and the success surface in
ways a test suite can't anticipate from the diff alone.** Each
new code path before the first network call is a candidate
for: (a) `meta.source` provenance correctness, (b) `--dry-run`
contract honouring, (c) precise gate predicates that match
the actual execution semantics, not the document's
syntactic features.

### Single load-bearing lesson

**AST-aware gating beats string-aware gating, and the
correctness bar climbs as the gate gets more specific.**

The original M6 raw command used neither — it neither parsed
the GraphQL document nor checked any predicate. Round 1's
fix added a mutation gate; the gate keyed off `hasMutation`,
which is correct for the **opt-in** decision (the agent must
acknowledge that the document contains a mutation before it
can be sent). Round 5's fix tightened the **dry-run**
decision to `selectedOperationKind` — which is correct for
the **execution-time** decision (only the selected operation
runs; that's the one whose kind matters for "should this
write or read").

The two predicates are different and that's deliberate.
Round 5's catch was that they had been collapsed into one
("hasMutation triggers everything"); the v0.2 lesson is to
keep mutation-related decisions **scoped to the question**:

  - "Should I require `--allow-mutation`?" → document-wide
    (`hasMutation`). Safety-by-default.
  - "Should I write or just preview?" → execution-scope
    (`selectedOperationKind`). Precision.
  - "Should I refuse subscriptions?" → document-wide
    (`hasSubscription`). The HTTP transport can't carry them
    regardless of selection.

A future v0.2 add (e.g. complexity-budget pre-flight on raw)
will face the same pattern: define the question precisely,
then key off the matching slot of `RawDocumentAnalysis`. Adding
a new slot is preferred over re-purposing an existing one.

### Why M7's close arc ran 2 rounds (vs M6's 6)

M7 is **hardening + docs**, not new runtime surface. M6's
six-round arc was driven by gate-correctness bugs nesting:
each fix exposed a novel bug class on the new code path
(`meta.source` provenance, `--dry-run` contract gap, gate
predicate precision). M7 has no new code path in front of
`resolveClient`; the only runtime additions are tests
(snapshot suite + tarball E2E), which are themselves the
contract-pinning gate, not gated by anything.

The four M7 findings clustered in two patterns:

1. **Doc-source-of-truth drift** (P1 #1). The error-code list
   appeared in three places: `src/utils/errors.ts ERROR_CODES`,
   cli-design.md §6.5 (the contract), CHANGELOG, and
   output-shapes.md. The first two were correct; the M7-new
   docs drifted because they were written without
   cross-referencing the source of truth — author memory of
   what shipped, not the literal list. Codex caught the
   drift by comparing all four loci.
2. **Status-block drift** (P1 #2). README + CLAUDE.md status
   blocks said "v0.1 in progress; M7 next" — written before
   M7 shipped, never updated when the M7 commits landed.
   Both were stale-by-construction (the in-flight commits
   were the very thing flipping the status). The post-mortem
   commit is the natural place to flip the status — Codex
   caught the gap because the status said one thing while
   the version-bump commit (74e390d) said another.

The two P2s were both small precision fixes:
- README schema example used `monday schema item-set` (hyphen).
  The schema command expects dotted names (`item.set`); `--help`
  scraping a v0.1 user could spend a confused minute on this.
- Snapshot suite missing bulk + dry-run shapes. Deferred per
  Codex's note "not a tag blocker" — the four shapes the suite
  covers (single rich, single dry-run, clear, multi-set) span
  every distinct §6.4 mutation envelope pattern; bulk has
  per-command integration coverage.

### Single load-bearing lesson

**Cross-reference the source of truth at write time, not at
review time.** The error-code drift is the canonical case:
three places in the docs claimed to mirror
`src/utils/errors.ts ERROR_CODES`; two ended up reciting
warning codes (`column_token_collision`,
`pagination_cap_reached`, `stale_cache_refreshed`) as errors,
and omitting the v0.1-active codes (`daily_limit_exceeded`,
`concurrency_exceeded`, `ip_rate_limited`, `resource_locked`,
the two `dev_*` reservations). Each list was written from
memory of the contract rather than by literal copy from
`ERROR_CODES`.

The fix is not a discipline ask ("I'll be more careful next
time"). It's a structural ask: when a doc claims to mirror a
source-of-truth list, the doc's tests should fail if the lists
diverge. The v0.2 / v0.3 candidate is a unit test that loads
`ERROR_CODES` and `grep`s the doc to assert the doc lists
exactly that set. CHANGELOG drift is harder to gate (the
0.1.0 entry is frozen once tagged) but the same principle
applies: the next minor release's changelog entry should be
generated from a typed source.

### Tests as a regression gate (M7 close specifics)

M7's two new test files **are themselves the regression
gates** — not just supporting evidence:

- `tests/e2e/published-tarball.test.ts`: 6 checks against an
  extracted `npm pack` tarball. The bin-target check would
  fail loudly if a future commit renamed
  `dist/cli/index.js`. The dev-paths-leak check would fail
  if `files: ["dist", "README.md", "LICENSE"]` ever broadens
  to `["**/*"]` accidentally. The `--version` cross-check
  would fail if the bumped source `package.json` ever fell
  out of sync with the tarball-embedded copy (the canonical
  "I bumped the source but didn't run prepack" failure mode).
  The `--help` noun check would fail if a command silently
  de-registered. The schema-envelope check would fail if a
  bundled module imported a missing path during build.
- `tests/integration/envelope-snapshots.test.ts`: 40
  snapshots, one per shipped command. Any v0.2 PR that
  renames a `data` / `meta` / `warnings` field, drops a slot,
  or re-orders an enum value fails this suite — even if the
  per-command integration tests are updated in lockstep with
  the rename, because the rename forces a deliberate
  snapshot update too. The "renamed key, fixed all per-command
  tests, forgot to bump schema_version" failure mode is now
  closed: the snapshot diff shows the rename, and an agent
  reviewing the PR sees it as a contract decision rather
  than implementation churn.

The Codex-found doc drift was caught at review time, not at
test time, because the docs aren't currently parsed by any
test. That's the v0.2 candidate above — a doc-vs-`ERROR_CODES`
parity check would convert this from "Codex catches it" to
"CI catches it."

Same gate as M3 / M4 / M5a / M5b / M6: every M7 close commit
landed with at least one test. The fix commit (`5deec78`) is
documentation-only — the test that would have failed against
the pre-fix CHANGELOG / output-shapes.md doesn't exist yet
(the v0.2 doc-parity test described above is its proxy). The
release-prep nature of M7 ("polish, not new runtime") makes
this acceptable for the milestone close; a hardening pass
that introduced runtime code without a regression test would
not be.

### Predictions for v0.1 close (vs reality)

§19 M5b post-mortem said v0.1's close-arc finding density
should "trail off" with 1-2 pass-1 importants for M6 and M7.
M6 actually ran six rounds (3 P1s + 7 P2s + 2 P3s = 12
findings) — substantially more than predicted, driven by
the gate-correctness chain. M7 ran two rounds (2 P1s +
2 P2s = 4 findings) — closer to the prediction shape, and
none of the findings were runtime-correctness (all
documentation drift). The trail-off held for the actual
runtime-affecting work; M6 was a tail-end exception
because the raw-GraphQL surface was the most contract-
sensitive new feature in the v0.1 cluster.

The v0.1 close as a whole landed inside the predicted
bounds for review-surfaced runtime findings, with the M6
six-round arc as the single anomaly. The total Codex
review investment across M0–M7 (~10 review rounds + ~1
fix commit per round average) compares favourably to
the cost of fixing equivalent contract drift after a
release tag, which the M0 token-leak finding's
counterfactual already established.
